# Оценка: [строка 59] Анализ ТРУ за 365 дней

## 1. Карточка оценки
- Номер строки: 59
- Название: Анализ движения номенклатуры за 365 дней
- Краткая формулировка требования: отчёт по движениям ТРУ за скользящий/фиксированный год в разрезе контрагента, договора, документа движения, счёта учёта с группировкой по пунктам перечня ТРУ.
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: прозрачность закупок и движения ТРУ для контроля и аудита.
- Границы: отчёт в 1С по движениям регистров; вне scope — выгрузка в BI без ТЗ.

## 2. Статус реализации
- Полностью реализовано: не подтверждено
- Частично реализовано: вероятно (универсальные отчёты по оборотам без группировки по перечню ТРУ)
- Пояснение: ключевая сложность — связка номенклатуры с пунктами перечня (10) и производительность за год по полному объёму движений.

## 3. Подтверждённые факты
- Связи в чек-листе: 10, 25, 37, 38.
- Требование явно задаёт высокую детализацию и длительный период.

## 4. Открытые вопросы
- Источник «пункта перечня ТРУ»: реквизит номенклатуры, регистр, классификатор.
- Нужен ли скользящий год от даты отчёта или календарный.
- Объём данных (порядок миллионов строк движений) и допустимое время формирования.

## 5. Новые объекты
- При разрыве: вспомогательный регистр накопления/витрина с агрегацией по дням для ускорения.

## 6. Существующие объекты
- Регистры бухгалтерии/накопления, справочник номенклатуры, договоры, контрагенты.

## 7. Зависимости
- Внутренние: 10 (категоризация/перечни), 25 (ID потребности при трассировке — опционально), 37–38.

## 8. Ролевая модель
- Финансовый и закупочный контур; возможны ограничения по организациям.

## 9. Варианты реализации
- **Минимально инвазивный:** СКД с запросом к движениям за период и соединением с перечнем ТРУ; ограничения по отбору для снижения нагрузки.
- **Архитектурный:** регламентное заполнение витрины (срезы по дням/неделям), отчёт читает только витрину; отдельный контроль полноты связи с перечнем.

## 10. Класс сложности
- Минимально инвазивный: **L** (тяжёлый запрос + множество разрезов)
- Архитектурный: **XL** (витрина + регламент + контроль)

- Таймауты и блокировки при формировании на продуктиве; необходимость ночного режима.

## 13. Гипотезы
- Часть разрезов уже доступна в типовых отчётах по взаиморасчётам/закупкам без группировки по перечню.
- Вероятность доработок: высокая из-за производительности и НСИ.

## 14. Антипаттерны
Риск запросов в цикле и N+1 при детализации — при реализации избегать; заложено в резерв архитектурного варианта.

## 15. Допущения

## 16. Решение по оценке
- Рекомендованный диапазон: min 8.5 / likely 12.0 / max 17.5; архитектурный — min 15.0 / likely 20.5 / max 26.5 чел.-дней.
