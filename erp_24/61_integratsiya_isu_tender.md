# Оценка: [строка 61] Интеграция с ИСУ Тендер

## 1. Карточка оценки
- Номер строки: 61
- Название: Интеграция с ИСУ Тендер
- Краткая формулировка требования: передача лотов, загрузка сведений о победителе и ценах победителя.
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: исключить ручной перенос тендерных результатов в закупочный контур УТ.
- Границы доработки (scope / out-of-scope):
  - **Scope:** исходящая выгрузка структуры лотов/позиций, входящая загрузка победителя и цен, сопоставление ключей, протокол, регламент, права.
  - **Out-of-scope:** функциональность самой ИСУ Тендер, юридическая экспертиза процедур, изменение регламентов закупок вне 1С.

## 2. Статус реализации
- Полностью реализовано: нет.
- Частично реализовано: не подтверждено объектами/рутинами с привязкой к «ИСУ Тендер», лотам закупок и ценам победителя как единому контуру.
- Не реализовано: **да** — по результатам семантического поиска метаданных и поиска BSL по смыслу не выявлены специализированные объекты/модули интеграции с ИСУ Тендер (в отличие от других именованных систем).
- Не определено (требует проверки): формат обмена (файл/API), версия ИСУ, обязательные поля лота.
- Вероятно покрыто частично: низкая вероятность скрытого названия; при наличии кастомных обработок вне индексируемых имён — уточнить у внедрения.
- Требует проверки: наличие внешних выгрузок из форм лотов/ПЗ под другим именем.

**Пояснение статуса:** для именованных интеграций (пример: КИС КЭД) в коде присутствуют устойчивые идентификаторы модулей и справочников; аналогичного следа для **ИСУ Тендер** не получено.

## 3. Подтверждённые факты
- Подтверждено анализом метаданных (поиск по имени): объекты с вхождением «Тендер» в имени в доступной выборке **не найдены**.
- Подтверждено семантическим поиском по описаниям: релевантные совпадения не дали объектов обмена с ИСУ Тендер.
- Подтверждено наличием иных ИСУ-контуров в конфигурации (например, выгрузки в **ИСУ СИП** из отдельного документа отчётности) — это **не эквивалент** требованию 61.

## 4. Открытые вопросы / пробелы
- Спецификация полей лота и ответа по победителю.
- Частота и инициатор обмена (регламент vs по кнопке).
- Связь с существующей загрузкой цен победителя (строка 33) — единый вход или дублирование.

## 5. Новые объекты системы
- Обработка(и) выгрузки/загрузки, регистр соответствия внешних идентификаторов лотов/позиций, возможно HTTP-сервис или подсистема в составе каркаса внешних интеграций.

## 6. Существующие объекты, требующие доработки
- Документы/объекты плана закупок и лотов (по результатам связанных оценок 28–30, 33) — точки привязки команд обмена и заполнения цен.
- Справочник контрагентов, регистры цен поставщиков — при загрузке цен победителя.

## 7. Зависимости
- Внутренние: 28–30 (лоты и победитель), 33 (цены), 24 (ПЗ).
- Внешние: ИСУ Тендер, согласованный формат, тестовый контур.

## 8. Ролевая модель и доступ
- Роли на запуск обмена и просмотр протокола; запрет редактирования цен без успешной загрузки/валидации (по политике заказчика).

## 9. Варианты реализации
- **Вариант A (минимально инвазивный):** файловый обмен (XML/JSON) + регламентная обработка + регистр сопоставления ключей.
- **Вариант B (архитектурный):** встраивание в каркас внешних систем с очередями, повторами, мониторингом и раздельными потоками выгрузки/загрузки.

## 10. Класс сложности
- **Вариант A:** **L** — несколько объектов, маппинг, регрессия цен и заказов.
- **Вариант B:** **XL** — устойчивый контур с очередями и наблюдаемостью.

**Степень модификации:** средняя/высокая.

## 17. Проверка quality gates
- Gate 1: критерии приёмки согласовать (полнота полей лота/победителя/цен).
