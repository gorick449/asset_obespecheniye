# Оценка: [строка 14] Заявочные кампании (периоды, корректировки, правила отбора)

## 1. Карточка оценки
- Номер строки: 14
- Название: Ведение заявочных кампаний
- Краткая формулировка требования: отдельная сущность кампании с окнами планирования, корректировки до утверждения и правилами отбора потребностей.
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: управляемый сбор и корректировка заявок до фиксации плана потребности и выхода в закупки.
- Границы доработки (scope / out-of-scope): внешний «календарь кампаний» без связи с 1С — out-of-scope.

## 2. Статус реализации
- Полностью реализовано: нет
- Частично реализовано: не подтверждено объектами с именованием/назначением «заявочная кампания»
- Не реализовано: отдельный документ/справочник кампании закупочного цикла не обнаружен
- Не определено (требует проверки): возможная кастомизация под иным именем (без ключевых слов в метаданных)
- Вероятно покрыто частично: нет
- Требует проверки: сценарии на продуктиве (внешние регламенты)
- Пояснение статуса (1-3 предложения): поиск документов по шаблону имени «Кампан*» не дал объектов; найденные совпадения относятся к журналу интеркампани, что не эквивалентно заявочной кампании закупок. Процессный контур кампаний в формулировке ТЗ **не подтверждён** метаданными и точками входа.

## 3. Подтверждённые факты
- В списке документов по подстроке «Кампан» в имени **0** объектов (поиск по типу «Документы»).
- Семантический поиск по описанию по запросу про заявочные кампании возвращает преимущественно подсистемы планирования/обеспечения, но **не** выделяет объект кампании закупок.
- Подсистема «Обеспечение потребностей» присутствует (связанный контур УП).

## 4. Открытые вопросы / пробелы (если есть)
- Ведётся ли кампания вне 1С (Excel/SharePoint) при ручном вводе в заказы?
- Есть ли кастомный объект без слова «кампания» в имени (требуется бизнес-интервью и продуктивный обзор)?

## 5. Новые объекты системы
- Справочник или документ «Кампания» с периодами (планирование / корректировка / утверждение), статусом, правилами отбора (хранение или ссылка на схему компоновки).
- Регистр сведений периодов и/или версий кампании.
- Связь заявок/заказов с кампанией.

## 6. Существующие объекты, требующие доработки
- Документы цепочки потребности (в т.ч. `ЗаказНаВнутреннееПотребление`, `ПланЗакупок`) — добавить связь с кампанией, проверки окон, запрет редактирования вне окна.
- Обработка `ОбеспечениеПотребностей` — фильтры по кампании при необходимости.

## 7. Зависимости
- Внутренние: строки 15–17, 20–21 (заявки, план потребности, согласования).
- Внешние: регламент закупок предприятия, оргструктура.

## 8. Ролевая модель и доступ
- Роли: инициатор заявки, руководитель, служба снабжения, администратор кампании.
- Права: разделение по кампаниям/подразделениям; возможны RLS по подразделению.
- Риск влияния на профили: высокий при жёстких окнах.

## 9. Варианты реализации
- **Минимально инвазивный:** документ «Кампания» + регистр периодов + реквизит ссылки в существующих документах + ограничения по датам/статусу без сложного движка правил.
- **Архитектурный:** полноценный мастер кампании с правилами отбора (СКД), версионированием, интеграцией с уведомлениями (41) и мониторингом (52).

## 10. Класс сложности
- **Минимально инвазивный:** **L** — новый мастер-объект процесса, затрагивает несколько документов и проведение/статусы.
- **Архитектурный:** **XL** — правила отбора, версии, интеграция со всей цепочкой 15–22.

## 13. Гипотезы и варианты
- Гипотезы: кампания может частично заменяться периодом в `ПланЗакупок` — **не подтверждено** как эквивалент трём окнам и правилам отбора.
- Вероятность доработок при подтверждении разрыва: **высокая**.
- Что проверить: продуктив, интервью с владельцем процесса закупок.

## 16. Решение по оценке
- Вывод: **не реализовано** в подтверждаемой объектной модели; требуется проектирование кампании.
- Рекомендованный диапазон: **минимально инвазивный** — 11.5 / 17.5 / 24.0; **архитектурный** — 25.0 / 36.0 / 50.0 чел.-дней.
- Условия валидности оценки: утверждённая модель окон, состав обязательных правил отбора, перечень затрагиваемых документов.

## 17. Проверка quality gates
- Gate 1: границы процесса требуют уточнения у заказчика.
- Gate 2: подтвержден отсутствующий объект кампании в обнаруженной номенклатуре метаданных; полнота поиска ограничена именованием.
- Gate 3: высокие процессные и организационные риски.
- Gate 4: классы L и XL, диапазоны с шагом 0.5.
- Gate 5: при нахождении скрытого объекта-кампании оценка пересматривается (возможно снижение).
