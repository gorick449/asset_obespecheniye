# Оценка: [строка 34] Загрузка цен из внешних источников

## 1. Карточка оценки
- Номер строки: 34
- Название: Загрузка цен из внешних источников
- Краткая формулировка требования: механизмы обмена (файл, API, регламент), сопоставление номенклатуры и протокол ошибок; влияние на плановые ориентиры и сравнительный анализ цен (связь с 33, 35, 58).
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: актуализировать ценовые ориентиры и данные для аналитики закупок без ручного ввода по каждой позиции.
- Границы доработки (scope / out-of-scope):  
  **Scope:** доработка/настройка загрузки в регистры цен (номенклатура/поставщики), сопоставление, журнал ошибок, при необходимости регламент.  
  **Out-of-scope:** юридическая валидация внешних прайсов, согласование цен (строка 36), интеграции, не описанные заказчиком (формат, частота, аутентификация).

## 2. Статус реализации
- Полностью реализовано: нет (зависит от конкретных внешних источников организации).
- Частично реализовано: да (типовой контур загрузки из файлов и работа с внешними сервисами для прайсов в рамках конфигурации).
- Не реализовано: не определено без ТЗ по конкретным источникам.
- Не определено (требует проверки): полнота покрытия выбранных заказчиком каналов (только файл / 1С:Бизнес-сеть / произвольный API).
- Вероятно покрыто частично: да.
- Требует проверки: согласование форматов и ключей сопоставления с контрагентом/ИТ.
- Пояснение статуса (1-3 предложения): по коду подтверждены загрузка цен из Excel/табличных данных и обработка «Загрузка цен поставщика из внешних файлов», а также запросы к сервисам получения прайс-листов (1С:Бизнес-сеть). Универсальный произвольный API под каждый внешний прайс без ТЗ считается не подтверждённым.

## 3. Подтверждённые факты
- Есть модули установки цен с загрузкой из Excel: `УстановкаЦенКлиент.ЗагрузитьИзExcel`, `УстановкаЦенСервер.ЗагрузитьИзExcel`, загрузка цен из таблицы значений (подтверждено BSL-индексом).
- Есть обработка `ЗагрузкаЦенПоставщикаИзВнешнихФайлов` с созданием документа `РегистрацияЦенНоменклатурыПоставщика` (подтверждено BSL-индексом).
- Есть контур работы с внешними прайс-листами через сервис: `ТорговыеПредложенияСлужебный.ЗапросСервисаПолучениеПрайсЛистов` и связанные методы (подтверждено BSL-индексом).
- Подсистема «Цены и скидки» присутствует в метаданных (подтверждено семантическим поиском по описанию).

- При варианте «произвольный API»: возможны новый HTTP-сервис/обработка обмена, регистр сведений «протокол загрузки», справочник «соответствие полей» — уточняется ТЗ.

## 6. Существующие объекты, требующие доработки
- `Обработки.ЗагрузкаЦенПоставщикаИзВнешнихФайлов`, документ `РегистрацияЦенНоменклатурыПоставщика`, общие модули `УстановкаЦенСервер` / клиент.
- При использовании Бизнес-сети: модули `ТорговыеПредложения*` (настройка и сценарии).
- Регламентное задание для периодической загрузки (если нет типового покрытия сценария).

## 7. Зависимости
- Внутренние: виды цен поставщиков, номенклатура, соглашения; отчёт сравнительного анализа цен (строка 58).
- Внешние: формат файла/API, доступность сервиса, учётные данные.

## 8. Ролевая модель и доступ
- Роли на загрузку цен и изменение регистров цен поставщиков; при регламенте — отдельная роль «обмен».
- RLS: при включённых ограничениях — проверить доступ к номенклатуре/партнёрам.

## 9. Варианты реализации
- **Минимально инвазивный:** доработать/настроить существующую загрузку из файлов (Excel/текст), расширить сопоставление и протокол ошибок, при необходимости регламентное задание.
- **Архитектурный:** отдельный модуль обмена по HTTP(S) с очередью, версионированием файлов и расширенным журналом; маппинг полей вынести в НСИ.

## 10. Класс сложности

### Вариант «Минимально инвазивный»
- **Класс: M** — несколько объектов, логика сопоставления и устойчивость к ошибкам.
- Степень модификации: средняя.
- Обоснование: опора на типовые механизмы; основной риск — качество сопоставления и объём тестов на реальных файлах.

### Вариант «Архитектурный (API)»
- **Класс: L** — подсистема обмена, безопасность, журналирование, обработка сбоев.
- Степень модификации: высокая.
- Обоснование: проектирование контракта API, регламенты, повторяемость, регрессия по ценообразованию.

- Вывод: требование закрывается **частично** типовыми средствами; доработка объёма зависит от канала обмена.
- Рекомендованный диапазон:  
  - минимально инвазивный: **3.5 / 6.5 / 9.5** чел.-дн.;  
  - архитектурный (API): **8.0 / 13.5 / 20.0** чел.-дн.  
- Предпочтительный вариант при неизвестном API: начать с минимально инвазивного, зафиксировать gap по API отдельной итерацией.
- Условия валидности оценки: утверждённые форматы, объём номенклатуры, политика сопоставления.

## 17. Проверка quality gates
- Gate 1 (полнота требования): частично — нужен перечень источников.
- Gate 2 (подтверждение реализации): частично подтверждено BSL/метаданными.
- Gate 3 (риски): учтены сопоставление и доступность сервиса.
- Gate 4 (сложность): два варианта с диапазонами.
- Gate 5 (валидность): оценка не актуальна при смене канала или массовой миграции истории.
