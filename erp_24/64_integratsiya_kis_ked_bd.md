# Оценка: [строка 64] Интеграция с КИС КЭД.БД

## 1. Карточка оценки
- Номер строки: 64
- Название: Интеграция с КИС КЭД.БД
- Краткая формулировка требования: получение данных о **заключённых договорах** и отражение в учётной системе (связь с договорами контрагентов и заказами).
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: актуальные договорные реквизиты и связь с закупками без ручного переноса из КИС.
- Границы доработки (scope / out-of-scope):
  - **Scope:** доработки по сравнению с критериями приёмки заказчика (виды договоров, обязательные реквизиты, массовые пакеты, связь с заказом поставщику).
  - **Out-of-scope:** функциональность КИС КЭД.БД; юридическое согласование формулировок договоров.

## 2. Статус реализации
- Полностью реализовано: **не подтверждено** без критериев приёмки заказчика по полноте сценариев.
- Частично реализовано: **да** — подтверждён развитый контур: договорные документы КИС КЭД, формирование/обновление `ДоговорыКонтрагентов`, регистр статусов, клиентские команды, переопределяемые процедуры по контрагентам и графикам.
- Не реализовано: нет как отсутствие механизма.
- Не определено: покрытие всех типов закупочных договоров и массовых пакетов, требуемых заказчиком.
- Вероятно покрыто частично: специфические сценарии (несколько контрагентов в одном ДД, сложные графики) — зависят от переопределяемых модулей.
- Требует проверки: автоматическое проставление договора в **заказах поставщику** (строка 37) для всех сценариев.

**Пояснение статуса:** по BSL виден сквозной сценарий «ДД КИС КЭД → договор в 1С → статусы/протокол»; для статуса «полностью» нужны согласованные критерии и проверка связки с заказами.

## 3. Подтверждённые факты
- Подтверждено BSL-кодом: функция `СформироватьДоговорНаОснованииДД` в `лтхИнтеграцияКИС_КЭД` формирует договор контрагента на основании элемента справочника `лтхПроектДоговорногоДокументаКИС_КЭД`, ведёт регистр сведений статусов (`лтхСтатусыПроектовДоговорныхДокументовКИС_КЭД` в описании), поддерживает перезаполнение и отдельные транзакции.
- Подтверждено BSL-кодом: `ОтобразитьНаФормеДоговораИнформациюОбИнтеграции`, `ПолучитьПолномочияТекущегоПользователяВЧастиИнтеграции` — UI и права на действия с ДД.
- Подтверждено BSL-кодом: `НайтиДоговорыПодчиненныеДоговорномуДокументу` в `лтхИнтеграцияКИС_КЭДВызовСервера` — поиск договоров, созданных на основании ДД.
- Подтверждено BSL-кодом: переопределяемые процедуры для поиска контрагентов по ИНН/КПП при неудаче по КССС, копирование графика платежей и др.

## 4. Открытые вопросы / пробелы
- Все ли виды договоров из практики заказчика проходят через текущий контур без ручных доработок.
- Требуется ли массовая загрузка пакетов ДД и каков SLA.

## 5. Новые объекты системы
- При **варианте B** возможны дополнительные обработки пакетной загрузки или расширение регистров мониторинга — по итогам аналитики.

## 6. Существующие объекты, требующие доработки
- `лтхИнтеграцияКИС_КЭД`, `лтхИнтеграцияКИС_КЭДПереопределяемый`, формы договоров, регистр статусов ДД.
- При необходимости: документы заказа поставщику — автозаполнение договора.

## 7. Зависимости
- Внутренние: 37 (заказы), 35 (цены/договорные условия при необходимости).
- Внешние: КИС КЭД.БД, канал доставки ДД.

## 8. Ролевая модель и доступ
- Уже заложены полномочия просмотра сканов, открытия ДД в КИС, смены статусов — проверить соответствие матрице ролей заказчика.

## 9. Варианты реализации
- **Вариант A (минимально инвазивный):** точечные доработки по результатам сравнения с критериями приёмки (заказ, реквизиты, отдельные типы ДД).
- **Вариант B (расширение):** пакетная обработка, дополнительные типы договоров/сторон, углублённый мониторинг и отчёты расхождений.

## 10. Класс сложности
- **Вариант A:** **S** — локальные доработки при уже существующем ядре.
- **Вариант B:** **M** — несколько сценариев и объектов, выше риск регрессии.

**Степень модификации:** низкая (A) / средняя (B).

## 16. Решение по оценке
- Вывод: функциональный контур **в значительной степени реализован**; оценка дана на **остаточный** объём по приёмке.
- Рекомендованный диапазон:
  - Вариант A: **2.0 / 4.5 / 7.5** чел.-дней
  - Вариант B: **6.5 / 11.5 / 17.0** чел.-дней
- Условия валидности: после UAT при отсутствии пробелов вариант A может быть снижен до **0–1** чел.-дня (сопровождение), что фиксируется отдельно.

## 17. Проверка quality gates
- Gate 1: критерии приёмки по договорам и заказам.
- Gate 2: сильные факты по BSL зафиксированы; полнота сценариев — пробел.
- Gate 3: умеренные риски регрессии.
- Gate 4: два варианта, шаг 0.5.
- Gate 5: при смене формата ДД в КИС — пересмотр.
