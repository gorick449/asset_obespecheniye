# Оценка: [строка 63] Интеграция с КИС КССС

## 1. Карточка оценки
- Номер строки: 63
- Название: Интеграция с КИС КССС
- Краткая формулировка требования: получение/ведение **кодов централизованных НСИ** (КССС) для сопоставления справочников.
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: единые корпоративные идентификаторы в НСИ для обменов и отчётности.
- Границы доработки (scope / out-of-scope):
  - **Scope:** реквизиты/регистры кодов КССС, загрузка/обновление, поиск по коду, обязательность, интеграция с формами ключевых справочников.
  - **Out-of-scope:** изменение самой КИС КССС; мастер-данные вне согласования с владельцем НСИ.

## 2. Статус реализации
- Полностью реализовано: нет (покрытие **всех** требуемых справочников НСИ и сценариев не подтверждено целиком).
- Частично реализовано: **да** — подтверждены модули и функции работы с **кодом КССС** для организаций, контрагентов и обобщённо для элементов справочников; связь с интеграцией ИСУ ВГО и СЭД.
- Не реализовано: нет как «полное отсутствие механизма».
- Не определено: полный перечень справочников, для которых на площадке требуется обязательный КССС (номенклатура, перечни ТРУ и др.).
- Вероятно покрыто частично: для части объектов код может вестись только внешне без доработок форм.
- Требует проверки: регистр `ЛТ_КодыКССС` и связанные процедуры в контексте страхового обмена — пересекается ли с корпоративным требованием по всем НСИ.

**Пояснение статуса:** ядро механизма **поиска и возврата кода КССС** и сопоставления по коду присутствует в BSL; требование 63 по «централизованным кодам НСИ» в полном объёме зависит от охвата справочников и регламентов заказчика.

## 3. Подтверждённые факты
- Подтверждено BSL-кодом: функции `КодыКСССОрганизаций`, `КодыКСССКонтрагентов`, `КодКССС`, `ПолучитьКонтрагентовПоКодамКССС`, `ПолучитьОрганизацииПоКодамКССС` в общих модулях `лтхИнтеграцияИСУВГО` и переопределяемых модулях.
- Подтверждено BSL-кодом: `ПолучитьКодКССС`, `ПолучитьСсылкуПоКодуКССС`, `ИмяРеквизитаКодКССС` в контурах `лтхИнтеграцияСЭД` / `лтхИнтеграцияСЭДПовтИсп`.
- Подтверждено BSL-кодом: `ИмяРеквизитаКодКССС` и `ЗначениеКодаКСС` в переопределяемых модулях для поиска реквизита КССС у справочников.
- Подтверждено BSL-кодом: упоминание регистра сведений **`ЛТ_КодыКССС`** в модуле `ЛТ_ПроцедурыОбменаИСУСтрахование` (контекст обмена — уточнить для полноты НСИ).

## 4. Открытые вопросы / пробелы
- Должен ли КССС быть обязательным для **номенклатуры** и **пунктов перечня ТРУ** (строка 10)?
- Источник истины: загрузка из КИС vs ручное ведение с контролем.

## 5. Новые объекты системы
- При **варианте B** возможен единый регистр соответствий «объект ↔ код КССС ↔ источник» и обработка синхронизации — по результатам аналитики.

## 6. Существующие объекты, требующие доработки
- Переопределяемые модули `лтхИнтеграцияИСУВГОПереопределяемый`, `лтхИнтеграцияСЭДПереопределяемый` — расширение списка типов и правил.
- Формы справочников НСИ — обязательность, проверки, быстрый поиск по коду.

## 7. Зависимости
- Внутренние: строка 10 (перечни ТРУ), строка 60 (прочие обмены).
- Внешние: КИС КССС, регламент выдачи/обновления кодов.

## 8. Ролевая модель и доступ
- Ограничение права массовой перезаписи кодов; аудит изменений (по политике).

## 9. Варианты реализации
- **Вариант A (минимально инвазивный):** доработать переопределяемые процедуры и реквизиты для **недостающих** справочников, контроль заполнения на формах.
- **Вариант B (архитектурный):** централизованный **регистр соответствий** + регламент синхронизации + единая панель мониторинга расхождений.

## 10. Класс сложности
- **Вариант A:** **M** — несколько объектов и проверок, умеренный риск регрессии обменов.
- **Вариант B:** **L** — дополнительные объекты данных, регламенты, отчёты расхождений.

## 13. Гипотезы и варианты
- Гипотезы: для номенклатуры код уже хранится в отдельном реквизите — требуется проверка форм и отчётов.
- Вероятность доработок: **средняя** после инвентаризации справочников.
- Что проверить: перечень справочников в регламенте заказчика vs реализация в переопределяемых модулях.

## 14. Антипаттерны и вклад в оценку
Подтвержденные антипаттерны в текущей реализации не выявлены.

## 15. Допущения и ограничения
- КИС КССС предоставляет стабильный справочник кодов или API.
- Не требуется полная перестройка мастер-системы НСИ в рамках задачи.

## 16. Решение по оценке
- Вывод: требование **частично реализовано**; остаточный объём зависит от перечня НСИ заказчика.
- Рекомендованный диапазон:
  - Вариант A: **4.5 / 7.5 / 10.5** чел.-дней
  - Вариант B: **8.0 / 14.5 / 21.5** чел.-дней
- Условия валидности: утверждён список справочников и правила обязательности КССС.

## 17. Проверка quality gates
- Gate 1: критерии приёмки по охвату НСИ.
- Gate 2: факты подтверждены BSL; полнота охвата — пробел.
- Gate 3: риски регрессии обменов учтены.
- Gate 4: два варианта оценены, шаг 0.5.
- Gate 5: при расширении перечня справочников — пересмотр.
