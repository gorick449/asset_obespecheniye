# Оценка: [строка 60] Интеграция с ERP

## 1. Карточка оценки
- Номер строки: 60
- Название: Интеграция с ERP
- Краткая формулировка требования: обмен с головной учётной системой (состав объектов и направления — по процессу заказчика).
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: единые НСИ и/или документооборот между УТ и ERP без двойного ввода и расхождений.
- Границы доработки (scope / out-of-scope):
  - **Scope:** проектирование и реализация контура обмена под конкретную ERP (объекты, направление, частота, сопоставление ключей), регламенты, протокол, права.
  - **Out-of-scope:** выбор вендора ERP, настройка самой ERP, нефункциональная инфраструктура вне 1С (VPN, сертификаты) — только учёт в рисках.

## 2. Статус реализации
- Полностью реализовано: нет (термин «ERP» в требовании не привязан к одному объекту метаданных).
- Частично реализовано: да — подтверждена корпоративная подсистема интеграций с внешними системами и обмен с отдельной внешней системой (не эквивалент «ERP» в общем виде).
- Не реализовано: нет (как универсального «ERP-коннектора» под требование без уточнения).
- Не определено (требует проверки): да — какая именно ERP, перечень объектов и мастер-система по видам данных.
- Вероятно покрыто частично: инфраструктура обменов БСП/планы обмена может использоваться точечно.
- Требует проверки: сценарии заказчика (что ведётся в ERP vs в УТ), чтобы не дублировать контуры 61–64.

**Пояснение статуса:** по метаданным и BSL подтверждены механизмы внешних интеграций (в т.ч. учёт внешних систем, потоки обмена, подписки на события, типовой обмен данными) и отдельный контур к ИСУ ВГО через веб-сервис; этого недостаточно, чтобы считать требование «интеграция с ERP» закрытым без спецификации ERP и объектов.

## 3. Подтверждённые факты
- Подтверждено BSL-кодом и описаниями рутин: наличие общего модуляного контура «интеграция с внешними системами» (подготовка/выгрузка, отслеживание изменений в регистрах бухгалтерии и в документах без проводок, состояние интеграции по системам).
- Подтверждено BSL-кодом: реализован обмен с **ИСУ ВГО (SAP ICMR)** — процедуры выгрузки/загрузки, обработка, веб-сервис `LTX_ICMR_Request`; это **отдельная** внешняя система, не синоним «ERP» из формулировки требования.
- Подтверждено метаданными (индекс): зарегистрированных **HTTP-сервисов** в выборке не обнаружено; при этом веб-сервис для ICMR присутствует в графе модулей (см. факт выше).
- Подтверждено наличием типовых подсистем: обмен данными (планы обмена, XDTO), миграция/выгрузка-загрузка — как потенциальная база для интеграции с иной ИБ/ERP.

## 4. Открытые вопросы / пробелы
- Какая ERP (продукт, версия, роль мастер-системы по номенклатуре, контрагентам, договорам, остаткам, проводкам)?
- Нужен ли обмен в реальном времени, пакетами, односторонне или с обратной загрузкой статусов?
- Как увязать с уже реализованными контурами КИС КЭД, КССС, ИСУ (избежать дублирования и конфликтов ключей)?

## 5. Новые объекты системы
- При варианте «отдельный сервисный контур»: обработка/общие модули обмена, регистры очередей и протокола, возможно план обмена или HTTP-сервис — по результатам аналитики.
- При варианте «план обмена БСП»: настройка состава объектов и правил трансформации (может не требовать новых метаданных, только правила).

## 6. Существующие объекты, требующие доработки
- Подсистема интеграций с внешними системами (потоки, константы/функциональные опции, подписки) — расширение под новую внешнюю систему «ERP».
- При необходимости: типовые объекты НСИ и документов, участвующие в обмене (реквизиты внешних ключей уже частично обкатаны в других контурах).

## 7. Зависимости
- Внутренние: НСИ (строка 10), договоры (связь с 64), планы закупок/потребности при обмене плановыми данными.
- Внешние: команда ERP, спецификация формата (API/XDTO/файл), сетевая доступность, сертификаты.

## 8. Ролевая модель и доступ
- Роли на новые обработки/регистры протокола; разграничение запуска регламентов и просмотра ошибок обмена.
- RLS: при обмене по организациям — проверить параметризованные функциональные опции (аналогично другим внешним системам).

## 9. Варианты реализации
- **Вариант A (минимально инвазивный):** использовать типовой **план обмена / универсальный обмен** или расширить существующий каркас внешних систем новым «потоком» и регламентами под ERP.
- **Вариант B (архитектурный):** выделенный **API-слой** (HTTP/REST или очередь сообщений) с нормализованным контрактом, идемпотентностью и журналом; 1С как клиент/сервер по согласованию.

## 10. Класс сложности
- **Вариант A:** **L** — несколько подсистем, маппинг и регрессия по НСИ/документам; без готовой спецификации ERP объём и риск растут.
- **Вариант B:** **XL** — несколько контуров, протокол ошибок, версионирование контракта, нагрузочные и отказоустойчивые требования.

**Степень модификации конфигурации:** средняя (A) / высокая (B).

## 13. Гипотезы и варианты
- Гипотезы: для части данных обмен может быть закрыт существующими планами обмена без новых объектов — требует проверки по целевой ERP.
- Вероятность доработок: **высокая** до получения контракта и перечня объектов.
- Что проверить: актуальные планы обмена в эксплуатации, уже настроенные узлы, политика мастер-системы.

## 15. Допущения и ограничения
- Оценка дана для одной целевой ERP и согласованного набора объектов; расширение перечня увеличивает сроки.
- Инфраструктура и лицензии внешних систем не включены.
- Тестирование включает интеграционные прогоны на тестовых стендах заказчика.

## 16. Решение по оценке
- Вывод: требование **не может считаться полностью реализованным** без привязки к конкретной ERP и подтверждённого контура обмена; инфраструктурная база в конфигурации **частично** присутствует.
- Рекомендованный диапазон:
  - Вариант A: **8.5 / 14.5 / 21.0** чел.-дней
  - Вариант B: **14.0 / 25.5 / 37.0** чел.-дней
- Условия валидности оценки: получены спецификация формата, перечень объектов, правила ключей, сценарии ошибок и повторов; не меняется мастер-система в ходе проекта без пересмотра оценки.

## 17. Проверка quality gates
- Gate 1 (полнота): требование на уровне бизнеса задано, критерии приёмки — уточнить с заказчиком (объекты, задержка, полнота).
- Gate 2 (реализация): статус опирается на анализ BSL/метаданных; прямой «ERP-коннектор» не выделен.
- Gate 3 (риски): зависимости от внешней команды и контракта учтены.
- Gate 4 (оценка): классы и диапазоны по вариантам заданы, шаг 0.5 соблюдён.
- Gate 5 (валидность): при смене ERP или объёмов обмена оценка пересматривается.
