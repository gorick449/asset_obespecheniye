# Оценка: [строка 13] Автоматический подбор аналогов в документах

## 1. Карточка оценки
- Номер строки: 13
- Название: Автоподбор аналогов в документах
- Краткая формулировка требования: при оформлении ключевых документов обеспечения/закупки система подставляет допустимый аналог по правилам (остатки, приоритет, запреты).
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: ускорение обработки дефицита без ручного поиска замены.
- Границы: логика в документах 1С перечня из задачи; вне scope — изменение договорных условий с поставщиком.

## 2. Статус реализации
- Полностью реализовано: не подтверждено
- Частично реализовано: вероятно (точечные сценарии в производстве/списании)
- Требует проверки: полный перечень документов из связей 12, 15, 22, 37 и единые правила приоритета
- Пояснение: без подтверждённой модели из строки 12 оценка автоподбора условна; объём растёт при множестве документов и сложных правилах.

## 3. Подтверждённые факты
- В конфигурации существуют программные точки, связанные с подбором/заменой на аналог в прикладном коде (подтверждено анализом BSL по ключевым именам процедур).
- Зависимость от строки 12 зафиксирована в чек-листе задач.

## 4. Открытые вопросы
- Список документов-получателей автоподбора (заказ поставщику, заявка, резервирование и т.д.).
- Триггер: только при нулевом остатке или по явной команде пользователя.
- Нужно ли журналирование факта подбора для отчёта 51.

## 5. Новые объекты
- Регистр сведений «Правила автоподбора аналогов» (склад, категория, приоритет) при необходимости; подписка/общий модуль.

## 6. Существующие объекты
- Документы из связей 15, 22, 37; модули форм и проведения; данные строки 12.

## 7. Зависимости
- Внутренние: 12 (обязательно), остатки, резервы, возможно 40.

## 8. Ролевая модель
- Автоподбор не должен обходить запреты; роль может включать/отключать автозамену.

## 9. Варианты реализации
- **Минимально инвазивный:** общий серверный метод выбора аналога по приоритету РС + вызов из 2–3 ключевых документов.
- **Архитектурный:** движок правил (настройки по организации/виду документа), очередь кандидатов, учёт партий/серий, интеграция с отчётом эффективности.

## 10. Класс сложности

- Архитектурный: **L**
