# Оценка: [строка 25] Сохранение ID потребности при консолидации

## 1. Карточка оценки
- Номер строки: 25
- Название: Сохранение ID потребности при консолидации ТРУ в план закупок
- Краткая формулировка требования: при объединении одинаковых ТРУ из разных заявок в строки ПЗ сохранять идентификатор(ы) исходной потребности для трассировки и отчётов.
- Дата оценки: 23.03.2025
- Версия оценки: 1.0
- Цель бизнеса: сквозная аналитика от потребности к закупке (в т.ч. отчёт за 365 дней, стр. 59).
- Границы доработки (scope / out-of-scope): модель данных и алгоритмы в 1С; производительность отчётов — отдельная задача при необходимости.

## 2. Статус реализации
- Полностью реализовано: нет
- Частично реализовано: возможно за счёт механизма «Назначений» (типовой контур обособленной потребности)
- Не реализовано: явный реквизит «ID потребности» в анализируемом фрагменте метаданных документа «План закупок» не обнаружен
- Не определено (требует проверки): наличие скрытых/расширенных реквизитов в других частях ТЧ или в регистрах планирования
- Вероятно покрыто частично: «Назначение» в шапке плана и в заказе поставщику как связь с потребностью
- Требует проверки: таблица соответствий при слиянии строк
- Пояснение статуса (1-3 предложения): точечный поиск по XML «План закупок» не выявил реквизитов с именами, содержащими «Потребн», «Идентификатор», «Заявк» в очевидном виде; для полноты нужна сверка всей ТЧ и регистров. Типовое «Назначение» может закрыть часть трассировки, но не эквивалентно явной таблице ID при консолидации.

## 3. Подтвержденные факты
- В документе «План закупок» присутствует реквизит «Назначение» (тип — ссылка на справочник назначений) с пояснением о планировании закупки под назначение.
- В документе «Заказ поставщику» есть реквизит «Назначение» с пояснением про обособленную потребность.
- По фрагменту файла метаданных, проанализированному для ТЧ «Товары», не найдено вхождений имён атрибутов с подстроками «Потребн», «Идентификатор», «Заявк».

## 4. Открытые вопросы / пробелы (если есть)
- Полный список реквизитов ТЧ «Товары» документа «План закупок».
- Использование регистров планирования/обеспечения для хранения связи «строка плана — множество потребностей».

## 5. Новые объекты системы
- Регистр сведений «Соответствие строки плана закупок и потребностей» или ТЧ подчинённого регистра; либо расширение структуры строки плана набором ссылок/идентификаторов.

## 6. Существующие объекты, требующие доработки
- Документ «План закупок», обработки консолидации/заполнения, отчёты [строка 52, 59].

## 7. Зависимости
- Внутренние: [строка 15, 20, 22, 24]; отчёты 52, 59.
- Внешние: при выгрузке в ИСУ — формат полей идентификатора.

## 8. Ролевая модель и доступ
- Ограничение видимости связей по подразделениям; RLS при необходимости на регистре соответствий.

## 9. Варианты реализации — если релевантно и есть неопределённость
- Минимально инвазивный: использовать «Назначение» и доработать правила, чтобы при слиянии сохранять список исходных назначений/заказов в регистре соответствий без изменения ключевой строки ТЧ.
- Архитектурный: явный UUID/код строки потребности на всех этапах и нормализованная таблица соответствий при консолидации.

## 10. Класс сложности

- Степень модификации: средняя

## 15. Допущения и ограничения
- Допущение: «ID потребности» — стабильный идентификатор строки потребности из контура [строка 15–20].
- Ограничение: анализ не охватил все регистры планирования.

## 16. Решение по оценке
- Вывод: явное требование по ID при слиянии не подтверждено текущей структурой плана; ожидается доработка или углублённая проверка регистров.
- Рекомендованный диапазон: 6.5 / 9.5 / 12.5 и 12.0 / 16.0 / 20.0 чел.-дней.
- Условия валидности: определение формата ID и сценариев слияния.
