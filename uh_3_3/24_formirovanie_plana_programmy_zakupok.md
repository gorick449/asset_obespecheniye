# Оценка: [строка 24] Формирование плана/программы закупок

## 1. Карточка оценки
- Номер строки: 24
- Название: Формирование плана/программы закупок (группировка по категориям, перечням; доработка)
- Краткая формулировка требования: консолидация и оформление плана/программы закупок с группировкой по категориям и перечням ТРУ, с учётом возможной доработки под учётную политику и внешние выгрузки.
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: единый утверждаемый план закупок с нужными разрезами для управления и отчётности (в т.ч. вне ИСУ).
- Границы доработки (scope / out-of-scope): доработки внутри конфигурации Управление холдингом 3.3 по объектам плана закупок и выгрузкам; out-of-scope — разработка внешней ИСУ Снабжение и согласование формата на стороне контрагента без ТЗ.

## 2. Статус реализации
- Полностью реализовано: базовый контур «программа закупок + строки плана + регистр строк + помощник планирования» в типовой подсистеме корпоративных закупок.
- Частично реализовано: группировка по «перечням ТРУ» в смысле внешнего ID потребности/перечня и выгрузка в ИСУ Снабжение по требованию 62; явная кастомная «доработка» из формулировки требования без ТЗ не подтверждена как объём.
- Не реализовано: не выявлено по метаданным как отсутствующий объект плана закупок.
- Не определено (требует проверки): фактический формат выгрузки для ИСУ Снабжение в конкретной инсталляции.
- Вероятно покрыто частично: специфические правила группировки, не совпадающие с «Помощником планирования» и «Положением о закупках».
- Требует проверки: настройки констант (сквозное планирование, товарные категории) в рабочей базе.
- Пояснение статуса: подтверждено метаданными и справкой объектов «Программа закупок», «Строка плана закупок», регистр «Строки плана закупок», «Положение о закупках», «Товарные категории», общая команда «Помощник формирования плана закупок».

## 3. Подтверждённые факты
- Документ «Программа закупок» (синоним в метаданных — план закупок 223-ФЗ) агрегирует ссылки на документы «Строка плана закупок» в табличных частях «Строки плана закупок» / «Строки плана валютные».
- Документ «Строка плана закупок» содержит реквизит «Идентификатор строки плана закупок» и обширную ТЧ «Номенклатура» с аналитиками планирования, ЦФО, периодом потребности, признаками 223-ФЗ (ОКПД2 и др.).
- Регистр сведений «Строки плана закупок» фиксирует актуальные версии строк в разрезе организации, периода, признака инновационного плана и номера ГПЗ.
- Семантический поиск по метаданным выявил «Помощник формирования плана закупок» и использование «Положения о закупках» для автоматического формирования/актуализации строк при изменении плана потребности (описано в справке к «Строка плана закупок»).
- «Товарные категории» используются для укрупнённого планирования и, при включённой константе, для разделения номенклатуры по категориям при формировании строк.

## 4. Открытые вопросы / пробелы
- Требуется ли заказчику отдельная группировка именно по «пунктам перечня ТРУ» из НСИ (требование 10), выходящая за пределы ОКПД2/номенклатуры — проверить заполненность реквизитов в прикладных данных.
- Уточнить целевой пакет обмена с ИСУ Снабжение (требование 62) и необходимость доработки выгрузки под категории/перечни.

## 5. Новые объекты системы
- При ограничении задачи типовым функционалом: не требуются.
- При необходимости отдельного контура выгрузки под ИСУ: возможны новые обработка/HTTP-сервис/план обмена (уточняется ТЗ на интеграцию).

## 6. Существующие объекты, требующие доработки
- Документы: «Программа закупок», «Строка плана закупок» (команды заполнения, отборы, печать/выгрузка).
- Регистры: «Строки плана закупок», при необходимости — связанные регистры ключевых дат/версий.
- Справочники: «Положение о закупках», «Товарные категории», номенклатура (реквизиты перечня ТРУ).

## 7. Зависимости
- Внутренние: заявки на обеспечение, план потребности, НСИ (перечни ТРУ), настройки констант.
- Внешние: ИСУ Снабжение (требование 62), при 223-ФЗ — ЕИС/обмен (уже заложен в типовой справке).

## 8. Ролевая модель и доступ
- Согласование программы закупок использует типовой механизм согласования (см. справку к документу); доработки ролей возможны при добавлении новых команд выгрузки или регистров.
- RLS: проверить внедрённые ограничения в конкретной базе.

## 9. Варианты реализации
- **Вариант А (минимально инвазивный):** настроить типовые правила в «Положение о закупках», константы, заполнение строк из потребности; без программирования или с точечными доработками отчётов/отборов.
- **Вариант Б (с интеграцией):** доработать/создать обмен с ИСУ Снабжение с маппингом категорий/перечней и протоколом ошибок.

## 10. Класс сложности
### Вариант А
- **S**, модификация низкая: используется типовой помощник и документы ПЗ без смены архитектуры.
### Вариант Б
- **M**, модификация средняя: несколько объектов обмена, сопоставление НСИ, тестирование стендов.

## 13. Гипотезы и варианты
- Гипотезы: часть формулировки «доработка» может означать только настройку, без кода.
- Вероятность доработок: средняя при обязательной интеграции с ИСУ; низкая при чисто типовом сценарии.
- Что проверить: макеты выгрузок, действующие константы, заполненность ОКПД2/перечней.

### 14. Антипаттерны и вклад в оценку
- Подтвержденные антипаттерны в текущей реализации не выявлены.

## 15. Допущения и ограничения
- Допущение: конфигурация соответствует типовой УХ 3.3 из индекса MCP.
- Ограничение: без прикладного ТЗ на ИСУ Снабжение оценка интеграции — диапазонная.

## 16. Решение по оценке
- Вывод: типовой функционал плана/программы закупок **подтверждён**; полное покрытие формулировки «перечни + доработка + ИСУ» зависит от интеграционного ТЗ.
- Рекомендованный диапазон: вариант А — **1.0 / 2.5 / 4.5** чел.-дн.; вариант Б — **4.5 / 7.0 / 9.5** чел.-дн.
- Условия валидности: уточнение требований 62 и 10, отсутствие скрытых изменений типовой логики в расширениях заказчика.

## 17. Проверка quality gates
- Gate 1 (полнота): выполнено на уровне оценки; критерии приёмки по перечням — уточнить.
- Gate 2 (подтверждение): факты из метаданных; интеграция — пробел.
- Gate 3 (риски): зафиксированы зависимости от НСИ и обмена.
- Gate 4 (сложность): классы и диапазоны заданы по вариантам.
- Gate 5 (валидность): допущения перечислены.
