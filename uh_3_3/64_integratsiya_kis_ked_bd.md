# Оценка: интеграция с КИС КЭД.БД (заключённые договоры)

## 1. Карточка оценки
- **Номер строки:** 64
- **Название:** Интеграция с КИС КЭД.БД (получение данных о заключённых договорах)
- **Краткая формулировка требования:** Входящее получение из КИС КЭД.БД сведений о заключённых договорах с отражением в учётной системе и использованием в закупках и ценообразовании.
- **Дата оценки:** 23.03.2026
- **Версия оценки:** 1.0
- **Цель бизнеса:** Актуальные договорные рамки без ручного ввода; опора для заказов поставщику и анализа цен.
- **Границы доработки (scope / out-of-scope):**
  - **Scope:** создание/обновление **ДоговорыКонтрагентов** и/или **ВерсияСоглашенияКоммерческийДоговор**; связь с контрагентом и организацией; сопоставление с заказами (требование 37) и ценами (35).
  - **Out-of-scope:** функционал самой КЭД.БД; юридическая валидация текста договора вне 1С.

## 2. Статус реализации
- **Полностью реализовано:** нет (как интеграция с КИС КЭД.БД)
- **Частично реализовано:** да
- **Не реализовано:** коннектор к КИС КЭД.БД и маппинг полей под регламент заказчика
- **Не определено:** формат обмена с КЭД.БД
- **Вероятно покрыто частично:** создание договоров из закупочного процесса (протокол победителя), обмен с ЕИС по контрактам, ЭДО
- **Требует проверки:** нужно ли создавать только карточку договора или полный график/спецификацию
- **Пояснение статуса:** Подтверждена развитая модель договоров (**ДоговорыКонтрагентов**, **ВерсияСоглашенияКоммерческийДоговор**) и сценарии закупок (в справке **ПротоколВыбораПобедителей** — создание договора по лоту). Есть типовые механизмы загрузки из внешних систем (**НастройкиЗагрузкиДанныхВнешнихСистем**, **ОбменДаннымиСВнешнимиСистемамиБП**, **ВыгрузкаЗагрузкаEnterpriseData**). Объектов «КЭД.БД» в метаданных не найдено.

## 3. Подтверждённые факты
- Подтверждено метаданными: справочник **ДоговорыКонтрагентов** с видами в т.ч. «С поставщиком»; документ **ВерсияСоглашенияКоммерческийДоговор** для параметров договора.
- Подтверждено метаданными: документ **ПротоколВыбораПобедителей** — сценарий создания договора после закупки; документ **ОтчетПоДоговорамЗаключеннымЗаказчикомПоРезультатамЗакупки_ЕИС** — учёт заключённых договоров для выгрузки на ЕИС (иной внешний контур).
- Подтверждено метаданными и BSL: справочник **НастройкиЗагрузкиДанныхВнешнихСистем** с менеджером **ЗагрузитьДанные** — загрузка данных, переданных внешней системой через сервис.
- Подтверждено поиском по коду: процедуры загрузки договоров из внешних источников в выдаче связаны преимущественно с ЕИС/типовыми паттернами, не с КЭД.БД по имени.

## 4. Открытые вопросы / пробелы
- Полный состав полей договора из КЭД.БД (суммы, валюта, срок, спецификация, признаки рамочного договора).
- Идентификация: ИНН/GLN/внутренний GUID КЭД.
- Обработка изменений и допсоглашений (версии договора в УХ).

## 5. Новые объекты системы
- При необходимости: регистр соответствия «Договор УХ — ИД КЭД»; журнал обмена.
- HTTP-сервис/обработка-адаптер — по транспорту.

## 6. Существующие объекты, требующие доработки
- **ДоговорыКонтрагентов**, **ВерсияСоглашенияКоммерческийДоговор** (заполнение из входящего сообщения).
- При связи с заказами: документы заказа поставщику (в объёме требования 37).
- Модули ЭДО при необходимости сопоставления с электронными документами (не эквивалент КЭД.БД без проверки).

## 7. Зависимости
- **Внутренние:** актуальные контрагенты и организации; настройки учёта по договорам.
- **Внешние:** КИС КЭД.БД, права доступа к API, тестовый контур.

## 8. Ролевая модель и доступ
- Роль загрузки договоров (ограниченный круг).
- Согласование изменений договоров — по существующим регламентам казначейства/закупок.

## 9. Варианты реализации
1. **Минимально инвазивный:** входящий файл/API → обработка сопоставления → создание/обновление карточки договора и версии; ручное дозаполнение спецификаций при сложном графике.
2. **Архитектурный:** полный маппинг спецификаций и графиков, автоматическое создание связанных документов, мониторинг расхождений с **ПротоколВыбораПобедителей**/заказами.

## 10. Класс сложности

### Вариант «Минимально инвазивный»
- **Класс:** M
- **Обоснование:** один основной объект (договор) и ограниченный набор полей при простой спецификации КЭД.
- **Степень модификации:** средняя.

### Вариант «Архитектурный»
- **Класс:** L
- **Обоснование:** версии, спецификации, связь с закупкой и заказами, контроль расхождений, высокий риск регрессии.
- **Степень модификации:** высокая.

## 13. Гипотезы и варианты
- **Гипотезы:** КЭД передаёт «шапку» договора; спецификация подтягивается позже.
- **Вероятность доработок:** средняя до утверждения полного сообщения.
- **Проверить:** нужна ли двусторонняя синхронизация.

## 14. Антипаттерны и вклад в оценку
Подтвержденные антипаттерны в текущей реализации не выявлены.

## 15. Допущения и ограничения
- Допущение: один юридически значимый ключ договора в сообщении КЭД.
- Ограничение: без ТЗ на спецификации графиков оценка архитектурного варианта верхняя граница может расти.

## 16. Решение по оценке
- **Вывод:** доработка требуется; готовой интеграции с КИС КЭД.БД не подтверждено, объектная база для договоров — есть.
- **Рекомендованный диапазон:**
  - Минимально инвазивный: **6.0 / 11.0 / 18.0** чел.-дней
  - Архитектурный: **13.5 / 23.0 / 35.0** чел.-дней
- **Условия валидности:** неизменность модели договора УХ; отсутствие необходимости переноса вложений и сканов из КЭД в объёме оценки.

## 17. Проверка quality gates
- Gate 1: нужна спецификация КЭД.БД.
- Gate 2: подтверждено метаданными — договоры и загрузка из внешних систем типовыми механизмами; КЭД.БД — нет.
- Gate 3: риски сопоставления зафиксированы.
- Gate 4: классы M и L.
- Gate 5: при полной спецификации с ТЧ — пересчёт в сторону архитектурного варианта.
