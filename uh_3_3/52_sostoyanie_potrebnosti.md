# Оценка: [строка 52] Отчёт «Состояние потребности» (стадии обеспечения)

## 1. Карточка оценки
- Номер строки: 52
- Название: Состояние потребности (на какой стадии обеспечения находится позиция)
- Краткая формулировка требования: сквозная видимость стадий от заявки до закрытия с опорой на ID потребности и связи с документами; корректность при частичном исполнении.
- Дата оценки: 23.03.2026
- Версия оценки: 1
- Цель бизнеса: контроль исполнения цепочки обеспечения и узких мест по стадиям.
- Границы доработки (scope / out-of-scope): отчётный контур и источники данных в 1С; не перепроектирование всего процесса [строки 15–23] без отдельного ТЗ.

## 2. Статус реализации
- Полностью реализовано: не подтверждено (нет проверки полного набора стадий и полей СКД).
- Частично реализовано: вероятно — в конфигурации есть несколько отчётов по потребностям и исполнению плана.
- Не реализовано: единый отчёт с названием/синонимом «состояние потребности» как единственная точка входа — не подтверждено.
- Не определено (требует проверки): покрытие всех стадий из требования и трассировка ID потребности в схемах компоновки.
- Вероятно покрыто частично: аналитика по удовлетворению/исполнению плана потребности.
- Требует проверки: регистр сведений «Состояния заявок на закупку» и прочие РС статусов в связке с отчётами.
- Пояснение статуса (1-3 предложения): По метаданным выявлены отчёты **АнализУдовлетворенияПотребностей**, **АнализПотребностей**, **ИсполнениеПланаПотребностиПоСрокам**, **ПотребностьВНоменклатуре** (у каждого есть макет **ОсновнаяСхемаКомпоновкиДанных**). Без разбора СКД нельзя утверждать полное соответствие формулировке «все стадии» и частичному выполнению.

## 3. Подтверждённые факты
- Объекты отчётов **АнализУдовлетворенияПотребностей**, **АнализПотребностей**, **ИсполнениеПланаПотребностиПоСрокам**, **ПотребностьВНоменклатуре** существуют; у **АнализУдовлетворенияПотребностей** в структуре метаданных отчёта нет реквизитов объекта (типовой отчёт на СКД), макет **ОсновнаяСхемаКомпоновкиДанных** присутствует — подтверждено анализом метаданных.
- В семантическом поиске по конфигурации фигурируют перечисления/регистры, связанные с состояниями заявок и исполнения (в т.ч. **СостоянияИсполненияЗаявки**, **СостоянияЗаявокНаЗакупку**) — подтверждено анализом метаданных (как кандидаты источников стадий).

## 4. Открытые вопросы / пробелы (если есть)
- Полный перечень стадий из регламентов заказчика vs поля и группировки существующих отчётов.
- Единый идентификатор потребности: заполненность и прохождение через консолидацию [строка 25].

## 5. Новые объекты системы
- При варианте 2: регистр сведений «Статус потребности / этап обеспечения» (если в текущих РС нет нужной детализации) или расширение существующих РС; вариант 1 — без новых объектов при достаточности СКД.

## 6. Существующие объекты, требующие доработки
- Один или несколько из: **АнализУдовлетворенияПотребностей**, **АнализПотребностей**, **ИсполнениеПланаПотребностиПоСрокам**, **ПотребностьВНоменклатуре** (схемы компоновки, варианты, параметры).
- При необходимости — документы/РС цепочки [строки 15–23] для обеспечения полноты данных.

## 7. Зависимости
- Внутренние: заявки, план потребности, заявки на закупку, план/программа закупок, заказы, поступления [строки 15–23, 37–38]; ID потребности [строка 25].
- Внешние: не подтверждены.

## 8. Ролевая модель и доступ
- Права на отчёт(ы) и расшифровки к документам; при многоорганизационном учёте — ограничения по организации/подразделению.

## 9. Варианты реализации — если релевантно и есть неопределённость
- **Вариант 1 — доработка СКД существующих отчётов:** объединить или расширить варианты, добавить стадии/расшифровки, опереться на текущие регистры.
- **Вариант 2 — нормализация статуса в РС + отчёт:** явные этапы с записью из подписок/проведения; отчёт только читает РС.

## 10. Класс сложности

## 17. Проверка quality gates.
- Gate 1 (полнота требования): стадии и сценарии частичного исполнения нужно зафиксировать с заказчиком.
- Gate 2 (реализация): подтверждены имена отчётов и макеты СКД; не подтверждено содержание стадий.
- Gate 3 (риски): зависимость от полноты процессных данных.
- Gate 4 (оценка): два варианта с диапазонами.
- Gate 5 (валидность): при отсутствии ID потребности в данных оценка варианта 1 растёт к верхней границе или переход на вариант 2.
