# Оценка: [строка 48] Интеграция с ЭДО для заполнения РНПТ из входящих счетов-фактур

## 1. Карточка оценки
- **Номер строки:** 48
- **Название:** ЭДО и автоматическое заполнение РНПТ из входящих СФ/УПД
- **Краткая формулировка:** перенос сведений о прослеживаемости/РНПТ из электронных первичных документов в поступление и партию.
- **Дата оценки:** 23.03.2025
- **Версия оценки:** 1.0
- **Цель бизнеса:** снизить ручной ввод РНПТ и ошибки при поступлении прослеживаемых товаров.
- **Границы:** типовой контур ЭДО и УПД в конфигурации; кастомные форматы вне 1С — отдельный проект.

## 2. Статус реализации
- **Частично реализовано / требует проверки сценария заказчика:** типовой мост **УПД ↔ поступление** присутствует (обработчики выбора УПД в формах поступления; модуль **ОбменСКонтрагентамиБП** содержит операции создания УПД) — подтверждено BSL-индексом (`ПоступлениеТоваровУслугФормы ОбработкаВыбораУниверсальныйПередаточныйДокумент`, `НайтиСоздатьУниверсальныйПередаточныйДокумент`).
- **Не подтверждено прямое вхождение строки «РНПТ»** в общих модулях с именем **ОбменСКонтрагентами** точечным поиском — перенос, вероятно, идёт через структуру УПД/НДС, а не отдельным текстовым полем в этих модулях.
- **Поступление товаров услуг** содержит логику учёта **РНПТ** и прослеживаемости в модуле объекта (множественные вхождения **РНПТ** в `ObjectModule`) — подтверждено BSL.

## 3. Подтверждённые факты
- Связь форм **Поступление товаров услуг** с **Универсальный передаточный документ** — подтверждено BSL `search_methods`.
- В модуле **ПоступлениеТоваровУслуг** есть проверки/параметры проведения с **РНПТ** — подтверждено BSL.
- Подсистема прослеживаемости и справочник/механизм **РНПТ** существуют — подтверждено метаданными.

## 4. Пробелы
- Полная карта полей УПД → ТЧ поступления для РНПТ без пошагового разбора форм (глубина одного сеанса RLM).
- Поддерживаемые форматы ЭДО (XML УПД версий) — уточнить по внедряемому оператору и релизу конфигурации.

## 5. Новые объекты
- Только при нестандартном операторе или нестандартном маппинге — см. вариант 2.

## 6. Существующие объекты
- **ПоступлениеТоваровУслуг**, **УниверсальныйПередаточныйДокумент**, **ОбменСКонтрагентамиБП**, подсистема ЭДО.

## 7. Зависимости
- **Внешние:** контур ЭДО, сертификаты, роуминг.
- **Внутренние:** 46 (признак прослеживаемости), 47 (проверка после заполнения).

## 8. Роли
- Права на загрузку ЭДО и редактирование поступления/РНПТ.

## 9. Варианты реализации
1. **Минимально инвазивный:** использовать загрузку УПД/входящего СФ штатным ЭДО, сопоставление номенклатуры штатными средствами, проверить на пилоте перенос РНПТ.
2. **Доработка маппинга:** расширение процедур заполнения поступления при разборе ЭДО (если на пилоте выявлены пробелы по РНПТ или комплектам).

## 10. Класс сложности
- **Вариант 1:** **S**
- **Вариант 2:** **M–L** (зависит от объёма форматов и точек интеграции)
