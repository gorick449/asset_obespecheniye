# Сводка по оценкам: допущения, ограничения, риски, пробелы, варианты (по группам ФТ)

Документ сформирован оркестратором на основе:

- перечня требований в `specs/requirements.md` (номера строк ФТ 10–64 и группы ФТ);
- файлов `1-asset_report/<N>_conststrains.md` и, для строк **14–23**, файлов `1-asset_report/<N>_constrains.md` (альтернативное имя в каталоге).

Ниже — **обобщение** по каждой группе ФТ (без дословного копирования всех пунктов по каждой строке). Детали по отдельному требованию — в соответствующих `<N>_conststrains.md` / `<N>_constrains.md` и отчётах `<N>_*.md`.

## Ведение НСИ (строки 10–11)

| Блок | Суммарно |
|------|-----------|
| **Допущения** | Конфигурация Управление холдингом (релиз/индекс как в оценках); изменения прав и обменов согласуются с владельцами интеграций; при работе через расширение — заложить резерв; для нормативов — единицы учёта номенклатуры, организационное согласование нормативов вне оценки, если не заявлено; доступность отчёта по строке 54 для доработок по нормативам. |

| **Варианты реализации** | **10:** усилить **КодВидаТРУ** + отчёты/выгрузки *vs* справочник перечня, связи, встраивание в СПЗ и обмены. **11:** расширить регистр + документ установки + отчёт *vs* модель с периодами/версиями и отдельными сценариями контроля. |

## Формирование потребностей (строки 14–23)

| Блок | Суммарно |
|------|-----------|
| **Допущения** | «Кампания» может быть реализована связкой оперативного плана, периодов и параметров закупки; заявка на потребность — **Оперативный план** / бюджет; корректировка — документ замены номенклатуры («Корректировка потребностей») при согласии заказчика; согласование заявки — **Заявка на обеспечение** (и при необходимости ОП); обработка потребностей — **Обеспечение потребностей**; план потребности — регистр **Планирование потребностей** и аудит версий; заявка на закупку — профиль ОП; согласование закупок — списки согласующих / роли БП. |
| **Ограничения анализа** | Нет полного прогона в продуктиве; не перечислены все регламентные задания; виды закупки из ТЗ не полностью сверены с перечислениями/ПВХ; цепочки ввода на основании и проведения — не до конца; блокировка СПЗ до утверждения и трассировка ID — частично в других строках (21, 25). |
| **Риски** | Терминология ТЗ ↔ объекты 1С; регрессия лимитов и оперативного планирования; дублирование документов; ошибки Excel; актуализация СПЗ после корректировки; несовпадение ролей «руководитель» и «эксперт ФН»; пересечение с типовым обеспечением ERP; регрессия отчётов при смене перечисления способов обеспечения; обход ограничений ручным вводом; миграция при вводе отдельного документа заявки; несовпадение оргмодели и ролей 1С. |
| **Чего не хватает / пробелы** | Явная сущность кампании с окнами и BSL-блокировками по всем точкам ввода; карта «вид закупки ТЗ → реквизит 1С»; подтверждение сохранения ID потребности на всех ветках (связь со строкой 25); доказательства блокировки включения в план без согласования; таблица «способ обеспечения → документ-результат»; подтверждение использования способа в проведении **Обеспечение потребностей**; отдельный документ «План потребности» как единственный носитель (vs регистр); оповещения (41) и стадии в отчёте (52); автоматическое включение строк в программу закупок без обходов; готовые маршруты с семью названиями служб. |
| **Варианты реализации** | **14:** донастройка периодов *vs* справочник/документ «Кампания». **15:** расширение аналитики *vs* новый документ. **18–19:** настройка и уточнение перечислений *vs* доработка проведения и отчётов. **22–23:** настройка маршрутов *vs* доработка кода маршрутов и отображения этапов в отчётах. |

## Отчёты (строки 51–59)

| Блок | Суммарно |
|------|-----------|
| **Допущения** | KPI аналогов сопоставимы с **УправлениеАналогами**; процессные регистры заполнены; ID потребности (25) технически доступен или добавляется без перестройки процесса; плановые даты — в заказе или плане поставок; ПЗ и план поставок используются в работе; КП и протоколы используются; пункт перечня ТРУ определим для номенклатуры/движения. |
| **Ограничения анализа** | Не полностью разобраны тексты СКД ряда отчётов (**ОтчетОПримененииАналогов**, **АнализЗаказовПоставщикам**, **АнализПоставок**, **ПланФактныйАнализЗакупок**, **АнализИсполненияПрограммыЗакупок**, **ОтчетГрафикПоставок**, **СравнениеПредложенийПоставщиков**); поиск отчётов СИЗ по маске не гарантирует полноту; смешение бух. и складских «резервов» в поиске (54); шум в метаданных по «ТРУ» (59). |
| **Риски** | Пустые отчёты без учёта замен (13); производительность на длинных периодах; несовпадение названий стадий с ожиданиями заказчика; дублирование и расхождение отчётов 55–57; зависимость сравнения цен от дисциплины КП (32); нагрузка на СУБД за год (59); разные толкования ДХНО и аварийного запаса (54). |
| **Чего не хватает / пробелы** | Формулы KPI «частота» и «эффективность» (51); матрица стадий и критерии закрытия строки (52); регистры выдачи СИЗ и обезличивание (53); регистры резервирования и правила «невостребованного» (54); первичный документ плана для план-факта (55); объект «план тендеров» и правила версии ПЗ (56); регламент даты факта поставки (57); требования к внешним прайсам для 58 (34); субконто/измерения перечня ТРУ и объёмы для выбора архитектуры (59). |
| **Варианты реализации** | Преобладают: доработка СКД *vs* регистр событий/витрина/ночной предрасчёт; для 55–57 — единая модель дат и статусов *vs* локальные правки одного отчёта. |

## Интеграция (строки 60–64)

| Блок | Суммарно |
|------|-----------|
| **Допущения** | **ERP:** заказчик даст перечень сущностей и направления; есть тестовый контур ERP. **ИСУ Тендер:** документированный интерфейс и тест; умеренные объёмы на пилоте. **ИСУ Снабжение:** спецификация формата и тест; выгрузка по утверждённым документам (21, 26 — в согласованном объёме); пакеты без отдельного high-load проекта. **КИС КССС:** список справочников и обязательность внешнего кода; стабильный справочник/API. **КЭД.БД:** структурированные данные или гибрид; в УХ достаточно контрагентов для сопоставления. |
| **Ограничения анализа** | ERP не конкретизирована в ТЗ; работы команды ERP и инфраструктура не входят; ИСУ Тендер не сопоставлен с одним объектом метаданных — опора на контур ЦУЗ–ЭТП/ЕИС; типовой обмен на ЕИС/ЭТП не заменяет 62 без решения заказчика; оценка КИС без спецификации — диапазонная; разработка на стороне КЭД.БД и юридическое признание ЭД договора вне оценки 1С. |
| **Риски** | Двойная запись мастер-данных с КИС/ИСУ; смена форматов ERP/ИСУ; задержка согласования формата и ключей ТРУ (62); расхождение статусов при частичном приёме (62); исторические данные без кодов КССС; коллизии номеров договоров и версий (64). |
| **Чего не хватает / пробелы** | Матрица объектов и частоты; правило мастер-системы (60); спецификация полей лота, победителя, цен; регламент обмена и ошибок (61); протокол, пример сообщения, обратный поток, SLA (62); модель данных КИС, частота обновления, удаления/переименования (63); XSD/JSON или пример пакета, график поставок, связь с лотом/закупкой (64). |
| **Варианты реализации** | **60:** типовой обмен *vs* адаптер под API ERP. **61:** расширение обработок площадок *vs* модуль-адаптер с журналом. **62:** обработка + РЗ + журнал *vs* план обмена + мониторинг + повторы. **63:** реквизиты + загрузка + проверки *vs* регистр соответствий + адаптер + отчёты расхождений. **64:** импорт шапки + журнал *vs* полный контур с версиями и связью с заказами. |
