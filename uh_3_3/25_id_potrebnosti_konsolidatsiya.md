# Оценка: [строка 25] Сохранение ID потребности при консолидации ТРУ в план закупок

## 1. Карточка оценки
- Номер строки: 25
- Название: Сохранение ID потребности при консолидации ТРУ в план закупок
- Краткая формулировка требования: при слиянии одинаковых ТРУ из разных заявок сохранять идентификатор исходной потребности и обеспечить трассировку для отчётов (в т.ч. 365 дней).
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: сквозная трассировка от заявки/потребности до строки плана и далее по цепочке закупки.
- Границы: доработки в документах/регистрах плана закупок и связанных отчётах; out-of-scope — изменение внешней ИСУ без согласования.

## 2. Статус реализации
- Полностью реализовано: не подтверждено как «внешний ID потребности заказчика» на уровне строки ТЧ «Номенклатура» документа «Строка плана закупок».
- Частично реализовано: есть «Идентификатор строки плана закупок» в шапке «Строка плана закупок» и справочник «Идентификаторы строк планов закупок»; регистр «Строки плана закупок» хранит версии строк — это **идентификатор строки плана**, не обязательно ID исходной потребности из заявки.
- Не реализовано: отдельная таблица соответствий «несколько заявок → одна агрегированная строка» **не выявлена** в перечне реквизитов ТЧ «Номенклатура» по результатам структурного запроса.
- Пояснение: подтверждено метаданными (`ИдентификаторСтрокиПланаЗакупок`, `ИдентификаторыСтрокПлановЗакупок`, `СтрокиПланаЗакупок`); явная колонка «ID потребности внешней системы» в ТЧ не обнаружена в выборке типов ТЧ.

## 3. Подтверждённые факты
- В шапке документа «Строка плана закупок» присутствует реквизит «Идентификатор строки плана закупок».
- Справочник «Идентификаторы строк планов закупок» имеет реквизиты периода, организации, номера ГПЗ, признака инновационного плана.
- Регистр сведений «Строки плана закупок» включает ресурс/связь с идентификатором строки плана и версией.
- Справка к «Строка плана закупок» описывает подбор из плана потребности и связь с заявкой на обеспечение (качественно, на уровне справки).

## 4. Открытые вопросы / пробелы
- Где физически хранится связь «строка ТЧ номенклатуры СПЗ → документ заявки/строка потребности» после консолидации — требуется проверка BSL подбора и движений (не выполнялась в полном объёме в данной сессии).
- Требование отчёта 59 по трассировке: нужно сопоставить с фактическими запросами отчёта в конфигурации.

## 5. Новые объекты системы
- Регистр сведений «Соответствие агрегированных строк плана исходным потребностям» (пример) — при подтверждении пробела.
- Или расширение ТЧ «Номенклатура» регистрами/реквизитами для множественных источников.

## 6. Существующие объекты
- «Строка плана закупок», регистр «Строки плана закупок», справочник идентификаторов, отчёты аналитики закупок.

## 7. Зависимости
- Внутренние: заявки на обеспечение, план потребности, НСИ.
- Внешние: при внешнем ID — обмены.

## 8. Ролевая модель
- При добавлении регистра соответствий — права на чтение/запись для служб закупок и контролёров.

## 9. Варианты реализации
- **А:** использовать типовую связь через аналитики планирования/источник (если подтвердится в BSL как достаточная).
- **Б:** ввести регистр соответствий агрегат ↔ множество исходных документов потребности.
- **В:** хранить внешний строковый ID в отдельном реквизите/ТЧ с валидацией.

## 10. Класс сложности
- **А:** XS–S (если типовая связь достаточна).
- **Б:** M (новый регистр, заполнение, отчёты).
- **В:** S–M (зависит от объёма отчётности и обменов).

## 13. Гипотезы
- Возможно, часть трассировки уже обеспечивается аналитиками без отдельного «ID» в ТЧ — **требуется BSL-проверка**.

### 14. Антипаттерны
- Подтвержденные антипаттерны в текущей реализации не выявлены.

## 15. Допущения
- Под «ID потребности» понимается либо внутренняя ссылка на документ потребности, либо внешний ключ из ИСУ.

## 16. Решение по оценке
- Вывод: **пробел по явному хранению множественных исходных потребностей на агрегированной строке не снят** структурным анализом ТЧ; оценка вариантов Б/В как наиболее вероятных при жёстком требовании таблицы соответствий.
- Диапазоны: А **1.0 / 2.0 / 4.0**; Б **4.5 / 7.0 / 9.5**; В **2.5 / 4.5 / 7.0** чел.-дн.

## 17. Quality gates
- Gate 2: неполное подтверждение поведения при консолидации — зафиксировано как пробел.
