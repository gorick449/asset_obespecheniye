# Оценка покрытия: [строка 15] Заявка на потребность / на обеспечение

## 1. Карточка оценки
- Номер строки: 15
- Название: Документы «Заявка на потребность», «Заявка на обеспечение» (Excel, категоризация закупки)
- Краткая формулировка требования: фиксация потребности с загрузкой из Excel и видами закупки (долгосрочная, северный завоз, плановая, дополнительная, аварийная).
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: единообразный ввод потребностей для планирования и закупок.
- Границы: объекты УХ 3.3; без детального разбора всех ПВХ видов бюджетов в ИБ.

## 2. Статус реализации
- Полностью реализовано: нет
- Частично реализовано: да
- Не реализовано: —
- Не определено: полное совпадение пяти названий видов закупки из ТЗ с конкретными перечислениями/реквизитами.
- Пояснение: документ **Заявка на обеспечение** и сценарии **Оперативный план** (в т.ч. «заявка на закупку» в коде загрузки) покрывают ввод и Excel; явного дублирующего документа «Заявка на потребность» с таким синонимом в выборке метаданных не найдено.

## 3. Подтвержденные факты
- Подтверждено метаданными: документ **Заявка на обеспечение**; справка описывает табличную часть **Потребности**, ценообразование, экспертизу по функциональным направлениям.
- Подтверждено метаданными: реквизиты шапки заявки включают **Вид бюджета планирования**, **Вид бюджета БДР**, период (`НачалоПериода`, `ОкончаниеПериода`), **Документ планирования** (тип связи с документом планирования — в индексе указан как документ из контура резервирования/планирования).
- Подтверждено BSL и структурой обработок: загрузка из внешних файлов поддерживает профиль **Заявка на закупку** (обработки загрузки оперативных планов и данных из файлов).
- Подтверждено BSL: общий модуль **Планирование потребностей** содержит логику получения планов по «заявке на закупку», что связывает оперативное планирование закупок с регистром потребностей.

## 4. Открытые вопросы / пробелы
- Требуется проверка в метаданных/ПВХ: наличие ровно пяти видов закупки в формулировке ТЗ как отдельного перечисления или аналитики.
- Уточнить у заказчика: является ли «Заявка на потребность» синонимом **Оперативный план** по бюджету закупок.

## 5. Новые объекты системы
- При необходимости разделения: документ «Заявка на потребность» как отдельный тип или уточнение представлений существующих документов (без дублирования данных).

## 6. Существующие объекты, требующие доработки
- **Заявка на обеспечение** (реквизиты/ТЧ, загрузка).
- **Оперативный план** (виды бюджета, загрузка).
- Обработки **Загрузка данных из внешних файлов** / **Загрузка оперативных планов из внешних файлов**.

## 7. Зависимости
- Внутренние: цены для расценки потребности, лимиты, **Планирование потребностей**.
- Внешние: шаблоны Excel и согласование с заказчиком по видам закупки.

## 8. Ролевая модель и доступ
- Роли инициаторов ЦФО, категорийных менеджеров, закупок; права на загрузку из файлов.

## 9. Варианты реализации
- **Минимально инвазивный:** расширить перечисление/аналитику видов закупки и проверки в **Заявка на обеспечение** и загрузке.
- **Архитектурный:** выделить два документа с чётким жизненным циклом и наследованием в РН без дублирования.

- **Степень модификации:** средняя

- **Класс:** L
- **Обоснование:** влияние на цепочку планирования и отчёты.
