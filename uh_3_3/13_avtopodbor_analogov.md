# Оценка: [строка 13] Автоматический подбор аналогов в документах

## 1. Карточка оценки
- Номер строки: 13
- Название: Автоматический подбор аналогов в документах
- Краткая формулировка требования: при вводе/подборе номенклатуры в документах цепочки потребности и закупки система должна предлагать или подставлять замену по правилам (приоритет, остатки, запреты).
- Дата оценки: 23.03.2026
- Версия оценки: 1
- Цель бизнеса: снизить ручной труд и ошибки при замене позиций в заявках и закупочных документах.
- Границы доработки (scope / out-of-scope): подбор/подстановка в пользовательских документах; ведение правил — [строка 12]; отчётность по факту — [строка 51].

## 2. Статус реализации
- Полностью реализовано: нет.
- Частично реализовано: **да** — есть форма выбора аналога на базе регистра настроек замены с параметрами, связанными с ценообразованием заявок на потребность; есть обработка управления аналогами в контуре планирования; документ **ЗаменаНоменклатуры** поддерживает сценарий замены с полями аналога в табличной части.
- Не реализовано: подтверждённый автоподбор в модулях документов **ЗаявкаНаОбеспечение**, **ЗаявкаНаЗакупку**, **СтрокаПланаЗакупок**, **ЗаказПоставщику** (прямые вхождения не найдены).
- Не определено (требует проверки): иные документы и общие механизмы подбора товаров (динамические вызовы без строкового совпадения).
- Вероятно покрыто частично: использование формы **ФормаВыбораАналога** регистра через универсальные механизмы выбора.
- Требует проверки: полный перечень точек входа в прикладных сценариях заказчика.
- Пояснение статуса (1-3 предложения): Правила и коэффициенты замены централизованы в регистре, пользовательский выбор фиксируется в **АналогиНоменклатуры**, но «автоматический» подбор в широком наборе документов закупки/обеспечения в проверенных каталогах не подтверждён.

## 3. Подтверждённые факты
- В форме **ФормаВыбораАналога** регистра **НастройкаЗаменыНоменклатурыЗакупок** используется константа **ТипЦенДляРасценкиЗаявокНаПотребность** — подтверждено BSL (связь с контуром заявок на потребность).
- Обработка **УправлениеАналогами** при сохранении выбора обращается к **НастройкаЗаменыНоменклатурыЗакупок.СрезПоследних** для коэффициента и записывает **АналогиНоменклатуры** — подтверждено BSL.
- Документ **ЗаменаНоменклатуры** содержит табличную часть с реквизитами **НоменклатураАналог**, коэффициентов и сумм — подтверждено метаданными.
- По индексированному поиску по каталогам указанных документов вхождения подстроки «Аналог» **не обнаружены** — подтверждено анализом кода в границах соответствующих папок документов.

## 4. Открытые вопросы / пробелы (если есть)
- Требуется ли заказчику именно **безакцептная** подстановка строки при создании документа или достаточно интерактивного выбора из списка аналогов.
- Нужно ли распространять подбор на **ЗаявкаНаОбеспечение** и **ЗаказПоставщику**, если основной контур завязан на заявки на потребность.

## 5. Новые объекты системы
- **Вариант 2:** возможен регистр сведений «журнал применения аналогов» или подписка на события — только при требовании сквозной аналитики; в минимальном варианте не обязателен.

## 6. Существующие объекты, требующие доработки
- Формы документов **ЗаявкаНаОбеспечение**, **ЗаявкаНаЗакупку**, **СтрокаПланаЗакупок**, **ЗаказПоставщику** (и при необходимости смежные) — команды подбора, обработчики изменения номенклатуры.
- Общие модули подбора/обеспечения, если замена должна вызываться из общего API.

## 7. Зависимости
- Внутренние: **НастройкаЗаменыНоменклатурыЗакупок**, **АналогиНоменклатуры**, обработка **УправлениеАналогами**, документ **ЗаменаНоменклатуры**.
- Внешние: не подтверждены.

## 8. Ролевая модель и доступ
- Доработка прав на команды подбора в новых формах документов; профили закупок и инициаторов заявок.
- RLS: уточнить при ограничении видимости номенклатуры по организациям/СПП.

## 9. Варианты реализации — если релевантно и есть неопределённость
- **Вариант 1 — минимально инвазивный:** добавить в ключевые документы команды «Подобрать аналог» / обработчики «при изменении номенклатуры» с открытием существующей формы выбора из регистра или чтением среза настроек; опционально — автозаполнение одной строки при однозначном срезе.
- **Вариант 2 — архитектурный:** выделить общий серверный модуль «ПодборАналогов» с едиными правилами приоритета, учётом остатков и запретов; журналирование фактов для отчёта [строка 51]; единая точка вызова из документов и обработок.

## 10. Класс сложности
- **Вариант 1:** **M** — несколько форм документов и согласованная логика вызова без перестройки архитектуры.
- **Вариант 2:** **L** — новый контур API, возможный регистр накопления/журнал, регрессия по планированию и закупкам.

- Вывод: функциональность **частично** реализована; для соответствия формулировке «в документах» по указанным типам документов ожидается **остаточная разработка**.
