# Оценка: [строка 30] Ввод данных о победителе тендера

## 1. Карточка оценки
- Номер строки: 30
- Название: Лотирование и тендеры — ввод данных о победителе
- Краткая формулировка требования: фиксация контрагента, цен и условий по итогам процедуры; связь с лотом; загрузка из ИСУ Тендер или ручной ввод; связь с заказом и ценами (33, 37, 61)
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: перевести результат тендера в учётные документы и ценообразование без потери связи с процедурой
- Границы доработки (scope / out-of-scope):
  - **Scope:** объект результата тендера в 1С, обязательные реквизиты для заказа/цен, обмен с ИСУ Тендер, связь с лотом (28)
  - **Out-of-scope:** юридическое хранение протоколов вне присоединённых файлов; подписание ЭП в 1С, если не требуется

## 2. Статус реализации
- Полностью реализовано: нет
- Частично реализовано: да (типовые **коммерческие предложения поставщика** и **регистрация цен**; кастомный справочник **проект договорного документа КИС КЭД** с контрагентами и закупочными реквизитами; интеграция приёма договорных данных через веб-сервис)
- Не реализовано: единый документ «итоги тендера по лоту» с явной семантикой «победитель» и обязательной связью «лот — контрагент — цены — заказ»
- Не определено (требует проверки): существующие обработки загрузки из ИСУ Тендер (отдельно от КИС КЭД)
- Вероятно покрыто частично: победитель фиксируется при создании договора/ДД из КЭД
- Требует проверки: сценарий «только цены победителя без договора» (33)
- Пояснение статуса (1-3 предложения): в метаданных **не найден** самостоятельный объект с именем/синонимом «победитель тендера». При этом контур **КП поставщика → регистрация цен → заказ** типовой; кастомный **КИС КЭД** закрывает часть сценария «договор из внешней системы». Для формулировки требования 30 в связке с **лотом** остаётся **разрыв**.

## 3. Подтвержденные факты
- Документ **`КоммерческоеПредложениеПоставщика`** содержит реквизиты контрагента/поставщика, валюту, ТЧ **«Товары»** с ценами и связью с строками запроса — подтверждено структурой метаданных.
- Документ **`РегистрацияЦенНоменклатурыПоставщика`** (упоминается в справке метаданных к связанным объектам) используется для фиксации цен и может создаваться на основании коммерческого предложения — подтверждено текстом справки в метаданных.
- Справочник **`лтхПроектДоговорногоДокументаКИС_КЭД`** содержит **`КонтрагентыУчастники`**, закупочные номера, суммы, **`СтатусТендера`**, даты **`TENDER_DATE`** — подтверждено структурой метаданных.
- Общий модуль **`лтхИнтеграцияКИС_КЭД`** содержит экспортные процедуры разбора пакетов договорных документов и обмена статусами с КИС КЭД — подтверждено BSL (описания процедур/функций в индексе кода).

## 4. Открытые вопросы / пробелы (если есть)
- Является ли ИСУ Тендер источником **первичного** ввода победителя или только КЭД
- Нужен ли отдельный шаг утверждения результата до загрузки цен (33)
- Обязательные поля НДС, валюты, срока действия цен — уточнить по регламенту

## 5. Новые объекты системы
- Документ **«Результат тендера / определение поставщика»** с ссылкой на лот, победителя, ТЧ цен/номенклатуры, статусом утверждения
- При необходимости: регистр сведений **«Загрузка из ИСУ Тендер»** (журнал соответствий ключей)

## 6. Существующие объекты, требующие доработки
- **`Документы.КоммерческоеПредложениеПоставщика`** / **`ЗапросКоммерческихПредложенийПоставщиков`** — связь с лотом и командой «зафиксировать победителя»
- **`Документы.ЗаказПоставщику`** (типовой) — заполнение из результата тендера
- Обработки обмена (при наличии) — разбор пакета ИСУ Тендер

## 7. Зависимости

- Внешние: **ИСУ Тендер (61)**, возможно **КИС КЭД** (64) как параллельный контур

## 8. Ролевая модель и доступ
- Роли: закупки, администратор интеграции, при необходимости юридическая служба (только чтение)
- Права: новый документ, изменение связанных объектов, просмотр журнала загрузки
- RLS: по организации
- Риск влияния на профили: средний

## 9. Варианты реализации — если релевантно и есть неопределенность
- **Минимально инвазивный:** расширить **КП поставщика** (признак «победитель», ссылка на лот) + типовая **регистрация цен** + ручное создание заказа
- **Архитектурный:** отдельный документ результата тендера с проведением в регистры согласованных цен/поставщика по лоту, регламентная загрузка из ИСУ, контроль обязательных полей до заказа

## 10. Класс сложности

- Обоснование степени модификации: в основном доработка типовых объектов и один новый документ при архитектурном варианте
