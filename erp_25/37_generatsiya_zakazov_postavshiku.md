# Оценка: [строка 37] Генерация заказов поставщику

## 1. Карточка оценки
- Номер строки: 37
- Название: Генерация заказов поставщику на основании планов закупок или вручную
- Краткая формулировка требования: формирование заказов из плана/программы закупок и/или ручной ввод; связь с договором и ценами; влияние на план-факт поступления и график поставок.
- Дата оценки: 23.03.2026
- Версия оценки: 1.0
- Цель бизнеса: исполнить утверждённый план закупок и зафиксировать условия поставки для последующего поступления и аналитики.
- Границы доработки (scope / out-of-scope):  
  **scope:** проверка покрытия типовыми объектами УП, настройка сценариев планирования, при необходимости стыковка с кастомным контуром ПЗ (требование 24), договорами (64), ценами (33–35).  
  **out-of-scope:** полная реализация тендерного контура (28–33), оценка отчётов 55–57 как отдельных задач.

## 2. Статус реализации
- Полностью реализовано: нет (зависит от фактического объекта «плана закупок» в эксплуатации и настроек сценария).
- Частично реализовано: да (типовой документ и рабочие места формирования).
- Не реализовано: не установлено.
- Не определено (требует проверки): идентичность корпоративного ПЗ документу «План закупок» и заполненность сценария флагом создания заказов поставщику.
- Вероятно покрыто частично: связь заказа с внешним ID строки потребности из требования 25 — без отдельной проверки реквизитов цепочки.
- Требует проверки: блокировки по статусу утверждения ПЗ (26) в живой базе.
- Пояснение статуса (1-3 предложения): В метаданных есть документ «Заказ поставщику», обработка «Формирование заказов поставщику по плану» и общая команда с тем же назначением; в справке документа заказа указаны ручное создание и автоматическое формирование по потребностям и по планам. Связь с план-фактом опосредована регистрами заказов и отчётом исполнения плана закупок. Если утверждённый план ведётся вне документа «План закупок», потребуется мост или обмен.

## 3. Подтвержденные факты
- В конфигурации присутствует документ «ЗаказПоставщику» с реквизитами «Договор», «Соглашение», «Партнер», «Контрагент», датами отгрузки/поступления, табличной частью «Товары» с полями «Назначение», «ИдентификаторСтроки», ценами и НДС — подтверждено структурой объекта в метаданных.
- Описание в справке к документу «Заказ поставщику» фиксирует возможность создания вручную и автоматического формирования обработками «Формирование заказов по потребностям» и «Формирование заказов по планам» — подтверждено текстом справки метаданных.
- Обработка «ФормированиеЗаказовПоставщикуПоПлану» со справкой о пошаговом формировании заказов из утверждённых планов закупок по сценарию с флагом «Создавать по сценарию: Заказы поставщику» — подтверждено метаданными и семантическим поиском.
- Обработка «ФормированиеЗаказовПоставщикам» (упрощённое обеспечение) создаёт заказы поставщику, в т.ч. для обеспечения заказов на отгрузку, с упоминанием заказов на перемещение — подтверждено справкой метаданных.
- В менеджерском модуле обработки «ФормированиеЗаказовПоставщикам» есть экспортная функция «СоздатьЗаказы», формирующая проведённые «ЗаказПоставщику» — подтверждено сигнатурой и описанием в метаданных BSL.
- Отчёт «ИсполнениеПланаЗакупок» описан для контроля плана, факта и непоступивших заказов — подтверждено метаданными отчёта.
- Регистр накопления «РасчетыСПоставщикамиПланПоставок» (плановые поставки) присутствует — подтверждено метаданными.

## 4. Открытые вопросы / пробелы (если есть)
- Совпадает ли прикладной «план/программа закупок» заказчика с документом «План закупок» и утверждением в сценарии планирования — не подтверждено.
- Явно ли в цепочке заказ→поступление сохраняется внешний «ID потребности» (требование 25) в полях, пригодных для отчёта «Состояние потребности» (52) — не подтверждено в рамках этой оценки.
- Настроены ли статусы заказов поставщику и их влияние на ожидаемые поступления для графика поставок (57) — требует проверки в базе.

## 5. Новые объекты системы
- При типовом покрытии: новые объекты не обязательны.
- Если план закупок ведётся вне 1С: возможны объекты обмена/обработки загрузки плановых строк — уточнять у заказчика.

## 6. Существующие объекты, требующие доработки
- Документ «ЗаказПоставщику» — при необходимости дополнительные реквизиты связи с внешним ПЗ/ID потребности.
- Сценарии планирования, виды планов — настройка флагов и периодичности.
- При интеграции КИС КЭД.БД (64) — согласование заполнения договора в заказе.

## 7. Зависимости
- Внутренние: документ «План закупок», способы обеспечения, соглашения и цены поставщика, опции «Заказы поставщикам», «Статусы заказов поставщикам».
- Внешние: требования 24–26 (ПЗ), 33–35 (цены), 55–57 (отчёты), 64 (договоры из КЭД).

## 8. Ролевая модель и доступ
- Роли чтения/изменения планов закупок и создания заказов (например, «ЧтениеПлановЗакупок», профили закупок) — проверить внедрение.
- Новые реквизиты — обновить роли и при необходимости условия RLS.

## 9. Варианты реализации — если релевантно и есть неопределенность
- **Минимально инвазивный:** включить сценарий с «Заказы поставщику», настроить обработку «Формирование заказов поставщику по плану», регламентировать ручной заказ; обучение пользователей.
- **Архитектурный:** если ПЗ в кастомном документе — разработать загрузку/преобразование в источники плана или API, контроль лимитов и дублирования заказов, журнал ошибок.

## 10. Класс сложности
- S / M / L / XL: **M**
- Обоснование класса: при чисто типовом сценарии объём невелик; при расхождении с кастомным ПЗ или обязательной трассировкой ID потребности растёт число объектов и тестов.
- Степень модификации конфигурации: средняя при доработках связи с внешним планом; низкая при одной настройке.
- Обоснование степени модификации: типовая цепочка уже в конфигурации; риск — предметная модель ПЗ заказчика.
