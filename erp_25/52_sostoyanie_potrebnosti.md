# Оценка: [строка 52] Отчёт «Состояние потребности»

## 1. Карточка оценки
- **Номер строки:** 52
- **Название:** Отчёты — Состояние потребности (на какой стадии обеспечения находится)
- **Краткая формулировка требования:** мониторинг строки/заявки потребности по стадиям цепочки обеспечения от заявки до закрытия.
- **Дата оценки:** 23.03.2026
- **Версия оценки:** v1
- **Цель бизнеса:** единое представление «где сейчас» по каждой потребности для снабжения и руководителей.
- **Границы доработки (scope / out-of-scope):**  
  - *Scope:* отчёт СКД (или расширение существующего), источники — регистры/документы контура заявок, плана потребности, ПЗ, заказов, поступлений, назначений; при необходимости — вспомогательный РС «стадия/срез».  
  - *Out-of-scope:* изменение самих бизнес-процессов согласования, внедрение нового типа закупочного контура без отдельного ТЗ.

## 2. Статус реализации
- **Полностью реализовано:** нет
- **Частично реализовано:** да (типовые средства мониторинга обеспечения и планов, не эквивалентны формулировке «состояние потребности» по кастомной цепочке)
- **Не реализовано:** отдельный отчёт с полным набором стадий под требования 15–23, 37–40 и трассировкой ID потребности (25)
- **Не определено (требует проверки):** есть ли кастомный отчёт/обработка с синонимом «потребность» вне индексируемых синонимов метаданных
- **Вероятно покрыто частично:** объединение типового отчёта «Запасы и потребности» / рабочих мест обеспечения с доработкой отборов
- **Требует проверки:** полнота стадий в данных при реальных статусах согласований и обменах
- **Пояснение статуса:** по метаданным видны регистр **ЗапасыИПотребности**, обработка **СостояниеОбеспеченияЗаказов**, отчёты плановой потребности производственного контура; прямого объекта отчёта с синонимом «состояние потребности» по заявочной цепочке не подтверждено.

## 3. Подтверждённые факты
- В конфигурации присутствуют объекты, связанные с **запасами и потребностями** и **обеспечением потребностей** (в т.ч. регистр **ЗапасыИПотребности**, обработка **СостояниеОбеспеченияЗаказов**, роль **ОтчетыМенеджераПоОбеспечениюПотребностей**).
- Отчёты с именем, содержащим «потребност», в категории «Отчеты» по шаблонному поиску сводятся преимущественно к **плановой потребности** производственного планирования, а не к мониторингу заявочной цепочки закупок.

## 4. Открытые вопросы / пробелы
- Где в данных фиксируется «стадия» по каждому шагу из требований 17–23 (отдельные статусы, БП, РС)?
- Обеспечивается ли связь строки отчёта с **ID потребности** (строка 25) на уровне запросов?
- Нужен ли единый справочник/перечисление стадий для согласованности с отчётами 56, 57.

## 5. Новые объекты системы
- При минимально инвазивном варианте: возможен только отчёт СКД без новых объектов, если измерения в существующих регистрах достаточны.
- При архитектурном: регистр сведений «Состояние/этап обеспечения потребности» или расширение существующих регистров с заполнением из подписок на проведение.

## 6. Существующие объекты, требующие доработки
- Регистры движений по обеспечению, документы заказов/поступлений, возможные кастомные документы заявок (подтвердить по внедрению).
- Настройки СКД и роли доступа к новым отчётам.

## 7. Зависимости
- **Внутренние:** 15–25, 37–40 (полнота и связность данных).
- **Внешние:** при обменах — не терять ключи связи строки потребности.

## 8. Ролевая модель и доступ
- Чтение отчёта: роли снабжения, при необходимости — ограничение по подразделениям/организациям.
- Риск: разные роли видят разные стадии — закладывается в тестирование.

## 9. Варианты реализации
1. **Минимально инвазивный:** один отчёт СКД на базе существующих регистров + внешние соединения к документам статусов; усечённая модель стадий, если часть этапов только в БП.
2. **Архитектурный:** нормализованное хранение стадии строки потребности (РС), заполнение событиями/подписками; отчёт как срез по РС + расшифровки.

## 10. Класс сложности
- **M** (при подтверждённых источниках стадий); **L**, если стадии размазаны по БП без единого среза.
- **Степень модификации:** средняя (отчёт + возможно небольшой РС или доработка запросов).
