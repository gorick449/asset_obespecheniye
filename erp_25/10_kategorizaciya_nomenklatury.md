# Оценка доработки: Категоризация номенклатуры

## 1) Карточка оценки
- Requirement ID: строка требования 10
- Название: Категоризация номенклатуры (по группам ТРУ, перечням)
- Группа функциональных требований: Ведение НСИ
- Дата оценки: 2026-03-19
- Версия оценки: v1

## 2) Краткая формулировка требования
Обеспечить категоризацию элементов `Справочники.Номенклатура` по группам ТРУ и перечням с поддержкой ввода/контроля в карточке и использования в прикладных сценариях НСИ.

## 3) Подтверждение текущей реализации (факты / гипотезы / пробелы)

### Подтвержденные факты (MCP)
- В `Справочники.Номенклатура` уже есть реквизит `КодТРУ` типа `Строка(255, Переменная)`.
- В `Справочники.Номенклатура` уже есть реквизит `ТоварнаяКатегория` типа `СправочникСсылка.ТоварныеКатегории`.
- В форме `Справочники.Номенклатура.ФормаЭлемента` есть контрол, привязанный к реквизиту `КодТРУ`.
- Справочник `Справочники.ТоварныеКатегории` существует (структурно минимальный: `Владелец`, `Родитель` и ТЧ `ДополнительныеРеквизиты`).
- Поиск по атрибуту `КодТРУ` показал вхождение в `Номенклатура` (узел категоризации уже частично присутствует в модели данных).

### Гипотезы/предположения
- Под «группами ТРУ» предполагается либо иерархический классификатор, либо набор нормализованных значений вместо свободного строкового `КодТРУ`.
- Под «перечнями» предполагаются отдельные справочники/регистры соответствия (например, перечни для бизнес-правил отбора/контроля).
- Логика контроля/заполнения может быть распределена между формой, модулем объекта и общими модулями НСИ.

### Пробелы (нужны данные для точной оценки)
- Нет формализованных критериев приемки (что считается «готовой категоризацией»).
- Нет перечня целевых сценариев использования (поиск, отбор, отчетность, обмен, контроль проведения).
- Не зафиксирован источник мастер-данных для ТРУ/перечней (ручной ввод, загрузка НСИ, интеграция).

## 4) Статус реализации
- Полностью реализовано: нет подтверждений полного покрытия требования.
- Частично реализовано:
  - хранение `КодТРУ`;
  - хранение ссылки на `ТоварнаяКатегория`;
  - наличие UI-поля `КодТРУ` в форме элемента.
- Не реализовано (предположительно):
  - формальная модель «групп ТРУ» и «перечней» как управляемых классификаторов;
  - полный набор правил валидации/автозаполнения/массового присвоения;
  - регламентированные проверки качества данных НСИ по категоризации.

## 5) Объекты системы
- Новые объекты (создать):
  - `Справочник/Регистр` для перечней категоризации (если отсутствует целевой объект под «перечни»).
  - (опционально) объект для нормализации «групп ТРУ», если строкового реквизита недостаточно.
- Существующие объекты (доработать):
  - `Справочники.Номенклатура` (реквизиты, проверки, возможно бизнес-правила).
  - `Справочники.ТоварныеКатегории` (при необходимости расширить структуру и атрибуты).
  - формы и модули, связанные с карточкой номенклатуры и массовой обработкой НСИ.
- Затрагиваемые объекты и модули:
  - `Справочники.Номенклатура` (метаданные + формы + модуль объекта/менеджера).
  - сопутствующие общие модули НСИ (точный перечень уточняется после фиксации сценариев).

## 6) Вариант реализации (рекомендуемый)
- Нормализовать категорийные данные (ТРУ/перечни) в управляемых объектах НСИ.
- Оставить `КодТРУ` как техполе только если нужен внешний код; иначе перейти к ссылочному хранению.
- Добавить правила валидации (обязательность, совместимость категории/вида номенклатуры, контроль дублей).
- Реализовать массовое заполнение/коррекцию и контроль качества данных перед промышленным использованием.

## 7) Риски и зависимости

### Антипаттерны и вклад в оценку
- Подтвержденные антипаттерны в текущей реализации не выявлены.
- Основание: в рамках подтвержденных фактов по требованию найдены только элементы метаданных (`КодТРУ`, `ТоварнаяКатегория`, контрол формы `КодТРУ`), без подтвержденных сценариев/кода, где уже проявляются антипаттерны.
- Вклад в оценку по разделу антипаттернов: **+0.0 чел.-дня**.

### Прочие риски
- Неоднозначность бизнес-семантики «перечней» (может существенно изменить модель данных).
- Возможная зависимость от интеграции/загрузки НСИ и владельцев справочников.
- Регрессионный риск в формах карточки номенклатуры и массовых сценариях редактирования.

## 8) Класс сложности
- S / M / L / XL: **M**
- Обоснование класса: изменение затрагивает несколько объектов НСИ, формы и бизнес-правила, но без подтвержденной кросс-подсистемной переработки.
- Степень модификации конфигурации: **средняя**
- Обоснование степени модификации: доработка существующей модели + вероятное добавление 1-2 объектов классификаторов и проверок.

## 10) Допущения и валидность оценки
### Допущения
- Требование ограничено контуром НСИ и карточкой `Номенклатура` без глубокой переработки смежных подсистем.
- Не требуется сложная интеграция с внешним MDM в рамках текущей задачи.
- Объем исторических данных допускает пакетную перекатегоризацию в регламентное окно.

### Когда оценка перестанет быть актуальной
- Появится требование на двустороннюю интеграцию/онлайн-синхронизацию перечней.
- Будут добавлены обязательные изменения в отчетность, права доступа и процессы согласования НСИ.
- Выяснится, что требуется миграция больших объемов исторических данных с контролем обратной совместимости.

## 11) Проверка quality gates
- Gate 1 (полнота требования): **частично** — бизнес-цель понятна, критерии приемки и границы не зафиксированы.
- Gate 2 (подтверждение реализации): **пройден частично** — подтверждены метаданные `КодТРУ`/`ТоварнаяКатегория`, форма `КодТРУ`; остальное требует уточнения сценариев.
- Gate 3 (риски и зависимости): **пройден** — технические риски и зависимости выделены.
- Gate 4 (сложность и трудозатраты): **пройден** — класс M, диапазон min/likely/max и резерв указаны.
- Gate 5 (валидность оценки): **пройден** — допущения и условия невалидности перечислены.
