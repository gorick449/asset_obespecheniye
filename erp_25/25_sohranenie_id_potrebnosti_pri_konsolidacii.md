# Оценка доработки: сохранение ID потребности при консолидации ТРУ в план закупок

## 1. Краткая формулировка требования

Бизнес-цель: обеспечить устойчивую трассировку от исходной строки потребности до строки `ПланаЗакупок` после консолидации одинаковых ТРУ, чтобы использовать эту связь в закупочном контуре и отчетах.

Критерии приемки:

- в системе есть идентификатор или иная устойчивая ссылка на исходную строку потребности;
- при объединении нескольких потребностей в одну строку `ПланаЗакупок` связь с каждым источником сохраняется;
- существует таблица соответствий или эквивалентный механизм "источник потребности -> строка плана закупок";
- связь может быть использована как минимум в требованиях 52 и 59.

Границы доработки:

- in scope: `Документы.ПланЗакупок`, `РегистрыНакопления.ПланыЗакупок`, серверное заполнение плана, модель хранения связи с источниками, влияние на отчеты 52 и 59;
- out of scope: отдельный анализ маршрута согласования плана закупок по требованию 26 и полный функциональный дизайн отчета 59 как самостоятельной задачи.

## 2. Статус реализации

**Не реализовано.**

В подтвержденном контуре закупочного планирования строки `ПланаЗакупок` формируются и переписываются по агрегированным аналитикам. Отдельный ID исходной потребности, таблица соответствий нескольких источников одной строке плана и подтвержденное использование такой связи в отчетах не выявлены.

## 3. Подтвержденные факты

- В конфигурации есть документ `Документы.ПланЗакупок` с табличной частью `Товары`.
- В `Документы.ПланЗакупок.Товары` подтверждены поля `Расшифровка` со строковым типом и `ДополнительныеПараметры` с типом `ХранилищеЗначения`, а также интервальные поля потребности `ИнтервалПотребностиОт`, `ИнтервалПотребностиДо`, `ЗаданИнтервалПотребности`.
- В `РегистрыНакопления.ПланыЗакупок` подтверждены измерения и ресурсы уровня агрегированной строки плана: `ПланЗакупок`, `Номенклатура`, `Характеристика`, `Назначение`, `ДатаЗаказа`, `Подразделение`, `Склад`, `Партнер`, `Соглашение`, `Сценарий`, `ВидПлана`, `Статус`, `Количество`, `Сумма`, `КЗаказу`, интервалы потребности.
- В `РегистрыНакопления.ПланыЗакупок` не подтверждено отдельное измерение, ресурс или реквизит для ID исходной строки потребности.
- Подтверждена точка входа серверного заполнения по дефициту: форма `Документы.ПланЗакупок.ФормаДокумента` вызывает серверную процедуру `ЗаполнитьПланыПоДефицитуНаСервере`, которая передает управление в `Планирование.ЗаполнитьПланыПоДефициту`.
- В `Планирование.ЗаполнитьПланыПоДефициту` строки результата загружаются напрямую в `ПланОбъект.Товары`.
- В `Планирование.СформироватьТекстЗапросаЗаполненияПоДефициту` поле `ДополнительныеПараметры` явно исключается из выборки пересчета, то есть не переносится в результирующую таблицу строк для консолидации.
- В том же контуре для колонок, не описанных отдельно, используется значение `NULL` на стороне расчета. Это не подтверждает сохранение структурированной ссылки на источник потребности для новых строк консолидации.
- Сопоставление существующих и пересчитанных строк в `Планирование.СформироватьТекстЗапросаЗаполненияПоДефициту` выполняется по агрегированным полям `Номенклатура`, `Характеристика`, `Назначение`, а не по идентификатору исходной строки потребности.
- В `Документы.ПланЗакупок.ТекстЗапросаТаблицаПланыЗакупок` движения в `РегистрыНакопления.ПланыЗакупок` формируются из строк документа по агрегированным полям строки плана без отдельного ключа источника потребности.
- По коду общего модуля `Планирование` не подтверждено использование имен `ИсходнаяПотребность` и `ИдентификаторЗаявки` в контуре заполнения `ПланаЗакупок`.
- По метаданным не подтверждено прямое использование `Документы.ПланЗакупок` и `РегистрыНакопления.ПланыЗакупок` в отчетах как ссылочного типа. Следовательно, подтвержденной прямой модели использования ID потребности в отчетах 52 и 59 на этом уровне нет.
- В `РегистрыНакопления.ПланыЗакупок` движения вносят `ПланЗакупок`, `КорректировкаРегистров` и `Сторно`, но это не дает отдельной трассировки до строки исходной потребности.

## 4. Гипотезы / предположения

- Текущий бизнес может частично опираться на непрямую идентификацию по сочетанию `Номенклатура + Характеристика + Назначение + период`, но такая схема неустойчива при консолидации одинаковых ТРУ из разных источников.
- Если отчет 59 строится по движениям закупочного контура, а не по явной цепочке "потребность -> план", требование строки 25 для него сейчас, вероятно, не покрыто.
- Наиболее жизнеспособная реализация потребует либо отдельного регистра соответствий, либо устойчивого технического ID строки потребности с поддержкой множественных источников.

## 5. Пробелы / что не подтверждено

- Не подтвержден объект-источник, в котором уже существует устойчивый ID строки потребности, пригодный для переноса в `ПланЗакупок`.
- Не подтверждена таблица соответствий "несколько строк потребности -> одна строка `ПланаЗакупок`".
- Не подтверждено использование этой связи в отчете 52 `Состояние потребности`.
- Не подтверждено использование этой связи в отчете 59 `Анализ ТРУ за 365 дней`.
- Не подтверждено сохранение трассировки после последующих этапов закупочного контура, если она будет нужна дальше уровня плана закупок.

## 6. Новые объекты системы

Вариант A:

- новые объекты не обязательны, если допустимо хранить только технический идентификатор или компактную ссылку в существующем контуре.

Вариант B:

- потребуется новый регистр сведений трассировки "строка потребности -> строка плана закупок";
- возможно потребуется технический идентификатор строки потребности в исходном объекте, если сейчас он неустойчив или отсутствует;
- при необходимости может понадобиться отдельный источник данных для отчетов 52 и 59.

- `Документы.ПланЗакупок`

- `ОбщийМодуль.Планирование`
- `РегистрыНакопления.ПланыЗакупок`
- смежный контур требований 52 и 59

## 8. Затрагиваемые объекты и модули

- `Документы.ПланЗакупок`
- `РегистрыНакопления.ПланыЗакупок`
- `ОбщийМодуль.Планирование`
- `Документы.ПланЗакупок.ФормаДокумента`
- процедуры серверного заполнения по дефициту
- механизм записи движений из `ПланЗакупок.Товары` в регистр

Процессный синтез текущего контура:

- точка входа: команда формы документа `ПланЗакупок`, ведущая к `ЗаполнитьПланыПоДефицитуНаСервере`;
- серверный механизм: `Планирование.ЗаполнитьПланыПоДефициту`;
- подтвержденные переходы данных: расчет дефицита -> загрузка строк в `ПланОбъект.Товары` -> запись движений в `РегистрыНакопления.ПланыЗакупок`;
- неподтвержденный элемент сценария: сохранение ссылки на одну или несколько исходных строк потребности после консолидации.

## 9. Роли, права, RLS

- Подтвержденный анализ метаданных и кода не выявил отдельного слоя прав или RLS, завязанного на ID потребности внутри текущего контура `ПланаЗакупок`.
- При варианте с отдельным регистром соответствий потребуется проверить права чтения для отчетов 52 и 59 и права записи в серверных процедурах заполнения плана.
- На текущем этапе влияние на ролевую модель оценивается как зависимость от выбранного варианта реализации, а не как уже существующий покрытый механизм.

## 10. Варианты реализации

### Вариант A: доработка существующего контура

Изменения:

- добавить устойчивый технический ID строки потребности в исходный контур;
- сохранять в строке `ПланаЗакупок` либо одиночный источник, либо сериализованный набор источников;
- доработать серверное заполнение и отчеты 52 и 59.

Плюсы:

- меньше новых объектов;
- быстрее внедрить в существующий контур;
- ниже входной порог для поддержки.

Риски:

- плохо масштабируется на сценарий "несколько источников -> одна строка плана";
- повышает риск хранения сложной связи в поле, не предназначенном для отчетного использования;
- выше риск регрессии при перепроведении и перерасчете.

Диапазон оценки:

- min / likely / max: **3.5 / 5.5 / 8.0 чел.-дня**

### Вариант B: отдельный регистр трассировки

Изменения:

- ввести отдельный регистр сведений соответствия источников потребности и строк `ПланаЗакупок`;
- поддержать запись и очистку этого регистра при каждом перерасчете и перепроведении;
- строить использование в отчетах 52 и 59 через устойчивую модель соответствий.

Плюсы:

- корректно покрывает множественную консолидацию;
- лучше подходит для аудита и аналитики;
- снижает зависимость отчетов от неструктурированных полей строки документа.

Риски:

- больше объем разработки и тестирования;
- потребуется контроль целостности при перепроведении и сторнировании;
- увеличивается сложность сопровождения.

Диапазон оценки:

- min / likely / max: **4.5 / 7.0 / 10.0 чел.-дня**

## 11. Риски, открытые вопросы и подтвержденные антипаттерны

Подтвержденные риски:

- функциональный разрыв: в текущем контуре нет подтвержденного хранения ID исходной потребности;
- риск потери трассировки при консолидации одинаковых ТРУ;
- риск неполного покрытия требований 52 и 59 без новой модели соответствий.

Открытые вопросы:

- нужен ли один идентификатор на строку источника или полноценная таблица множественных соответствий;
- нужно ли продолжать трассировку дальше уровня `ПланаЗакупок`;
- должен ли идентификатор быть видим пользователю или оставаться техническим.

Подтвержденные антипаттерны:

- подтвержденные антипаттерны производительности в текущей реализации не выявлены;
- подтвержден именно функциональный, а не производительный разрыв.

Вклад антипаттернов в оценку: **+0.0 чел.-дня**.

## 12. Сложность

Вариант A:

- класс сложности: **M**
- степень модификации: **средняя**

Обоснование:

- затрагиваются документ, серверный модуль планирования и отчетные зависимости;
- изменение локально по теме трассировки, но проходит через несколько объектов.

Вариант B:

- класс сложности: **L**
- степень модификации: **средняя**

Обоснование:

- появляется отдельная модель хранения соответствий;
- требуется интеграция с отчетным и серверным контуром перерасчета.

## 14. Допущения, условия валидности оценки и что проверить дополнительно

Допущения:

- анализируется основная конфигурация без скрытого внешнего механизма трассировки;
- требование ограничено уровнем плана закупок и отчетов 52 и 59;
- в смежном контуре нет уже реализованного отдельного регистра соответствий, не проявленного в подтвержденных данных.

Условия потери валидности оценки:

- если заказчик потребует протянуть ID потребности до заказа поставщику, поступления и последующих отчетов;
- если фактическая консолидация выполняется вне `ПланЗакупок` в другом объекте или внешней системе;
- если потребуется версионная история пересборки соответствий по корректировочным планам.

Что проверить дополнительно:

- точный объект-источник строки потребности и наличие в нем устойчивого ключа;
- бизнес-правило для сценария "две разные потребности -> одна строка плана";
- требуемую глубину использования связи в отчетах 52 и 59;
- необходимость переноса трассировки в требования 37, 55 и 57.

## 15. Проверка quality gates

- Gate 1. Полнота требования: **частично пройден**. Бизнес-цель понятна, но нужна точная граница использования ID.
- Gate 2. Подтверждение текущей реализации: **пройден**. Подтверждены серверный контур консолидации и отсутствие явной трассировки ID в подтвержденных данных.
- Gate 3. Риски и зависимости: **пройден**. Выявлены функциональные риски и зависимость от требований 52 и 59.
- Gate 4. Сложность и трудозатраты: **пройден**. Даны варианты реализации, классы сложности и оценки min / likely / max.
- Gate 5. Валидность оценки: **пройден**. Зафиксированы допущения, условия невалидности и дополнительные проверки.
