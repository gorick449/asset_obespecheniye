# Оценка: [строка 31] Формирование тендерной документации по шаблонам

## 1. Карточка оценки
- Номер строки: 31
- Название: Лотирование и тендеры — формирование тендерной документации по шаблонам
- Краткая формулировка требования: макеты шаблонов, заполнение из данных лота/ПЗ; набор типов документов; отличие от ручной подготовки вне 1С
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: сократить трудоёмкость подготовки типовых файлов тендерной документации и обеспечить согласованность с данными учёта
- Границы доработки (scope / out-of-scope):
  - **Scope:** печатные формы (MXL/офисные шаблоны), подстановка полей из лота/ПЗ/НСИ, каталог шаблонов, версионность при необходимости
  - **Out-of-scope:** ЭП и размещение на площадках ЕИС из 1С (если не запрошено отдельно); хранение оригиналов вне присоединённых файлов

## 2. Статус реализации
- Полностью реализовано: нет
- Частично реализовано: да (типовые механизмы печати и макеты у документов закупок и КП; интеграция **КИС КЭД** ориентирована на **договорные** документы, а не на полный комплект тендерной документации из лота)
- Не реализовано: библиотека шаблонов именно **тендерной** документации с привязкой к **лоту** и параметризацией всех требуемых видов документов
- Не определено (требует проверки): перечень обязательных видов документов по регламенту заказчика
- Вероятно покрыто частично: отдельные печатные формы из ПЗ без «лот-контекста»
- Требует проверки: использование внешних шаблонов Word через БСП
- Пояснение статуса (1-3 предложения): в метаданных **не выявлено** объекта «лот» и связанного набора шаблонов тендерной документации. Наличие **макетов печати** у документов коммерческих предложений подтверждено, но это **не эквивалентно** полному комплекту тендерной документации по лоту.

## 3. Подтвержденные факты
- У документа **`ЗапросКоммерческихПредложенийПоставщиков`** в метаданных указаны макеты **`ПФ_MXL_ЗапросНаКоммерческоеПредложение`** и сравнения условий — подтверждено списком макетов объекта.
- Документ **`ПланЗакупок`** содержит данные для подстановки в печатные формы (номенклатура, количества, партнёр и др.) — подтверждено структурой метаданных.
- Подсистема **КИС КЭД** обслуживает **договорные** документы (веб-сервис приёма ДД) — подтверждено метаданными и описанием модулей интеграции; это **другой** класс документов, чем тендерная документация этапа до договора.

## 4. Открытые вопросы / пробелы (если есть)
- Полный перечень шаблонов (ТЗ, извещение, проект договора, техническое задание и т.д.)
- Формат вывода: только MXL или Office Open XML
- Нужна ли сквозная нумерация и журнал сформированных комплектов

## 5. Новые объекты системы
- Справочник **«Виды тендерных документов»** и хранилище **шаблонов** (файлы + метаданные полей)
- Обработка **«Формирование комплекта тендерной документации»** с выбором лота/ПЗ

## 6. Существующие объекты, требующие доработки
- После появления **лота (28)** — источник данных для шаблонов
- **`Документы.ПланЗакупок`** — дополнительные поля для подстановки при отсутствии лота
- Общие модули печати БСП (при использовании) — расширение

## 7. Зависимости
- Внутренние: **28**, **24**, НСИ **10**
- Внешние: при необходимости — согласование с юридической службой (организационная)

## 8. Ролевая модель и доступ
- Роли: закупки (формирование), юрист (чтение/скачивание)
- Права: справочник шаблонов, обработка формирования, присоединённые файлы
- RLS: по организации
- Риск влияния на профили: низкий–средний

## 9. Варианты реализации — если релевантно и есть неопределенность
- **Минимально инвазивный:** 2–3 макета MXL + команда печати из обработки с параметрами отбора из ПЗ; без отдельного лота — временная привязка к документу ПЗ
- **Архитектурный:** каталог шаблонов, Office-шаблоны с тегами, пакетное формирование комплекта, версии, присоединение к лоту

## 10. Класс сложности
- S / M / L / XL: **M** (при 3–5 шаблонах); **L** при большом количестве видов документов и Office-интеграции
- Обоснование класса: в основном конфигурирование печати и обработки; растёт с числом шаблонов и требований к верстке
- Степень модификации конфигурации: **средняя** / **высокая** (для архитектурного варианта)
- Обоснование степени модификации: новые объекты хранения шаблонов и обвязка печати
