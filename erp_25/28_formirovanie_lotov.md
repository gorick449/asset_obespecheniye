# Оценка: [строка 28] Формирование лотов

## 1. Карточка оценки
- Номер строки: 28
- Название: Лотирование и тендеры — формирование лотов
- Краткая формулировка требования: формирование лотов из плана/программы закупок, правила объединения/разбиения, подготовка к выгрузке в ИСУ Тендер (связь с 24, 29–32, 61)
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: структурировать позиции ПЗ для закупочных процедур и обеспечить трассируемость при передаче во внешнюю тендерную систему
- Границы доработки (scope / out-of-scope):
  - **Scope:** модель лота в 1С, связь со строками ПЗ (документ `ПланЗакупок`), правила группировки/деления, идентификаторы для обмена с ИСУ Тендер (61), роли и отчётность по лотам
  - **Out-of-scope:** юридическая экспертиза 44-ФЗ; полная замена ИСУ Тендер; детальная спецификация формата обмена без ТЗ от владельца ИСУ

## 2. Статус реализации
- Полностью реализовано: нет
- Частично реализовано: да (источник строк — типовой документ планирования закупок; интеграционная подсистема под корпоративный контур закупок/договоров присутствует в конфигурации, но не как объект «лот»)
- Не реализовано: отдельная сущность лота с правилами лотирования и привязкой к ПЗ в явном виде
- Не определено (требует проверки): фактическая выгрузка лотов в ИСУ Тендер в промышленной среде (контракт полей, регламент)
- Вероятно покрыто частично: учёт закупки на стороне КИС КЭД/договорного контура без выделенного лота в 1С
- Требует проверки: наличие скрытых/переименованных объектов под «лот» вне стандартного поиска по имени
- Пояснение статуса (1-3 предложения): по структуре документа `ПланЗакупок` (табличная часть «Товары») не выявлено реквизитов уровня лота; поиск объектов метаданных по смыслу «лот/тендерная процедура» не выявил документа или справочника «Лот». Покрытие требования в текущем виде оценивается как **неполное**; реализация потребует проектирования новой сущности или расширения ПЗ.

## 3. Подтвержденные факты
- В конфигурации присутствует документ **`ПланЗакупок`** с табличной частью **`Товары`** (номенклатура, количество, партнёр, склад, даты заказа/поступления, цена и др.) — подтверждено структурой метаданных объекта.
- Отдельного объекта метаданных с назначением «лот закупочной процедуры» и связью «лот — строки ПЗ» по результатам поиска **не подтверждено**.
- В конфигурации есть кастомный контур интеграции **КИС КЭД** (веб-сервис договорных документов, общий модуль интеграции) — подтверждено метаданными и сигнатурами экспортных процедур/функций BSL; это **не эквивалентно** полноценному лотированию из ПЗ без дополнительной проработки.

## 4. Открытые вопросы / пробелы (если есть)
- Утверждённый формат и ключи сопоставления для ИСУ Тендер (внешний ID лота, позиции, версия ПЗ).
- Обязательность лотирования до/после утверждения ПЗ (связь с требованиями 24, 26).
- Нужны ли неделимые лоты, лимиты по сумме, запрет смешения категорий ТРУ — правила не видны в метаданных ПЗ.

## 5. Новые объекты системы
- Документ и/или справочник **«Лот закупки»** с табличной частью позиций и ссылками на источники в ПЗ (или регистр соответствия «строка ПЗ — лот»).
- При необходимости: перечисление **статусов лота**, регистр сведений для обмена с ИСУ Тендер.

## 6. Существующие объекты, требующие доработки
- **`Документы.ПланЗакупок`** — команды заполнения/отбора строк в лот, контроль повторного включения, связь с документом лота
- Подсистема обмена (существующие **HTTP/Web-сервисы**, **ир**-обработки при наличии) — расширение под выгрузку лотов
- **Роли** закупок и интеграции — право на формирование лотов и просмотр статусов выгрузки

## 7. Зависимости
- Внутренние: планирование закупок (`ПланЗакупок`, сценарии, статусы ПЗ — 24, 26), НСИ номенклатуры/перечней (10), при необходимости ID потребности (25)
- Внешние: **ИСУ Тендер** (61), согласование регламента обмена с владельцем системы

## 8. Ролевая модель и доступ
- Роли: менеджер по закупкам, администратор интеграции, при необходимости только чтение для контролёров
- Права: новый объект лота, изменение связанных форм `ПланЗакупок`, выполнение обработок выгрузки
- RLS: при внедрении RLS по организации/подразделению — распространить на лот и строки
- Риск влияния на профили: средний (новый объект + изменение закупочного рабочего места)

## 9. Варианты реализации — если релевантно и есть неопределенность
- **Минимально инвазивный:** регистр сведений «Лот» + строки в табличной части в обработке; ручное формирование; выгрузка в файл/JSON по кнопке; слабый контроль на уровне ПЗ
- **Архитектурный:** полноценный документ «Лот закупки» с проведением, блокировкой повторного включения строк ПЗ, статусами, подписями на события, регламентной выгрузкой в ИСУ Тендер, журналом ошибок

## 10. Класс сложности
- S / M / L / XL: **L**
- Обоснование класса: затрагивается несколько подсистем (план закупок, интеграция, возможно отчёты 56/58), нужна устойчивая модель данных и контроль целостности
- Степень модификации конфигурации: **высокая**
- Обоснование степени модификации: новые объекты и связи с существующим планом закупок плюс внешний контур

## 16. Решение по оценке
- Вывод: требуется **существенная доработка** для явного формирования лотов в 1С и увязки с ПЗ/ИСУ Тендер
- Рекомендованный диапазон: **11.0 / 17.5 / 27.5** чел.-дней
- Условия валидности оценки: не меняется модель ПЗ кардинально (например, отказ от документа `ПланЗакупок`); объём шаблонов обмена — не более одного основного и двух доработок формата

## 17. Проверка quality gates
- Gate 1 (полнота требования): частично — уточнить правила лотов и формат 61
- Gate 2 (подтверждение реализации): выполнено на уровне метаданных; отдельный объект лота не подтверждён
- Gate 3 (риски): интеграционная зависимость зафиксирована
- Gate 4 (сложность): класс L, диапазон указан
- Gate 5 (валидность): зависит от спецификации ИСУ Тендер и статуса ПЗ
