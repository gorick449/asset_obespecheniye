## 1. Карточка оценки
- Номер строки: 34
- Название: Ценообразование — загрузка цен из внешних источников
- Краткая формулировка требования: получение цен из внешних прайсов/систем (файл, API, регламент), сопоставление номенклатуры, учёт в ориентирах для анализа и закупки.
- Дата оценки: 2026-03-23
- Версия оценки: 1.0
- Цель бизнеса: расширить базу ценовых сигналов для сравнительного анализа (треб. 58) и планирования без ограничения только ручным вводом.
- Границы доработки (scope / out-of-scope):
  - **В scope:** настройка и при необходимости доработка загрузчиков, маппинг полей, протокол ошибок, права, связь с видами цен поставщика.
  - **Out-of-scope:** юридическая валидация внешних прайсов, разработка внешних маркетплейсов/поставщиков.

## 2. Статус реализации
- Полностью реализовано: нет (как универсальное требование «любой внешний источник»)
- Частично реализовано: да
- Не реализовано: нет
- Не определено (требует проверки): перечень конкретных внешних систем заказчика
- Вероятно покрыто частично: да
- Требует проверки: да
- Пояснение статуса (1-3 предложения): подтверждено метаданными наличие типовых механизмов загрузки и обновления цен, включая обработку **ЗагрузкаЦенПоставщикаИзВнешнихФайлов** и регламентные задания вроде загрузки с маркетплейсов и **ОбновлениеЦен**. Покрытие произвольного API конкретной внешней системы заказчика не подтверждено без спецификации протокола.

## 3. Подтвержденные факты
- Подтверждено метаданными: обработка **ЗагрузкаЦенПоставщикаИзВнешнихФайлов** — штатный контур многошаговой загрузки цен поставщика с этапами сопоставления номенклатуры.
- Подтверждено метаданными: наличие регламентных процедур обновления цен (в т.ч. сценарии для внешних каталогов/маркетплейсов в составе подсистемы цен).
- Подтверждено метаданными: регистр **ЦеныНоменклатурыПоставщиков** как целевое хранилище цен по партнёру и виду цены поставщика.
- Подтверждено метаданными: в справочнике **ВидыЦен** предусмотрены способы получения цен из внешних источников (в т.ч. маркетплейсы) в модели ERP 2.x.

## 7. Зависимости
- Внутренние: НСИ номенклатуры и партнёров; треб. 33 (цены победителя) и 35 (последняя закупка) как соседние источники для отчёта 58.
- Внешние: доступ к внешним системам, сертификаты, лимиты API.

## 8. Ролевая модель и доступ
- Роли загрузки цен и администрирования обмена; разграничение «только просмотр ориентиров» vs «проведение в цены».

## 9. Варианты реализации — если релевантно и есть неопределенность
- **Минимально инвазивный:** использование **ЗагрузкаЦенПоставщикаИзВнешнихФайлов** и типовых регламентов; настройка шаблонов и сопоставления без программирования или с точечными доработками формата.
- **Архитектурный:** отдельный коннектор под API заказчика (очередь, идемпотентность, мониторинг), запись с явным признаком источника и версии пакета.

## 10. Класс сложности
- S / M / L / XL: **S** (настройка типового контура) до **M** (новый API)
- Обоснование класса: при файловом/типовом сценарии трудозатраты низкие; при новом протоколе — средние.
- Степень модификации конфигурации (низкая/средняя/высокая): **низкая** / **средняя**
- Обоснование степени модификации: типовые объекты уже закрывают основную модель; риск роста модификации при нестандартном API.

## 13. Гипотезы и варианты

- Вероятность доработок: **низкая** при файловом контуре; **средняя** при уникальном API.
- Что проверить: список систем, частота обновления, объём номенклатуры.

## 15. Допущения и ограничения
- Допущение: внешний источник предоставляет хотя бы артикул/код и цену в согласованной валюте.
- Ограничение: оценка «архитектурного» варианта вынесена в `34_asset.md` и может доминировать при отсутствии файлового представления.

## 16. Решение по оценке
- Вывод: типовая конфигурация **частично** закрывает требование; доработка масштабируется от настройки до разработки коннектора.
- Рекомендованный диапазон: **2.0 / 3.0 / 5.5** чел.-дней (см. допущения).
- Условия валидности: утверждён перечень источников и способ обмена; при одном файловом источнике верхняя граница ближе к **3.0**.

## 17. Проверка quality gates
- Gate 1: выполнено по постановке из `tasks.md` и `design.md`.
- Gate 2: частичная реализация подтверждена метаданными обработки и регламентов.
- Gate 3: риски сопоставления и обмена зафиксированы.
- Gate 4: класс **S–M**, числа кратны 0.5.
- Gate 5: при появлении только типовых файлов — пересмотр в сторону **S** и снижение max.
