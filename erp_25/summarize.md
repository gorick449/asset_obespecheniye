## Ведение НСИ (строки 10–11)

| Тема | Сводка |
|------|--------|
| **Допущения** | Категоризация ТРУ может опираться на `КодТРУ` и кастомные `лтх_*` без отдельного объекта «перечень»; нормативы запасов — через `ТоварныеОграничения`; полная доработка всех отчётов/выгрузок ИСУ в этой итерации не обязательна. |
| **Ограничения** | Явный справочник «перечень ТРУ» в метаданных не подтверждён; сквозное использование классификации в заявках, ПЗ и обменах не проверено целиком; отдельный реквизит «аварийный запас» не выделен; вывод нормативов в отчёте по запасам (54) не верифицирован. |
| **Риски** | Расхождение кодировки ТРУ заказчика с `КодТРУ`; дублирование классификации в типовых и `лтх_*`; трактовка «аварийный» vs три уровня запасов; рассогласование отчёта 54 с ожиданиями. |
| **Чего не хватает** | Утверждённая структура групп/перечней ТРУ; обязательность `КодТРУ` в операционных документах; определение «аварийный запас»; проверка формы отчёта по запасам (54). |
| **Варианты реализации** | **Минимально:** усилить `КодТРУ` и `лтх_*`; использовать существующие уровни запасов. **Архитектурно:** отдельный справочник перечня ТРУ и соответствие номенклатуре; отдельный ресурс/реквизит «аварийный» и единые правила расчёта/отображения. |

| **Варианты реализации** | **Минимально:** объект кампании поверх плана; расширение `ЗаказНаВнутреннееПотребление`; доработка БП; расширение типовой корректировки/обработки; БП поверх плана; расширение существующих документов и БП закупки; сопоставление типовых перечислений способов. **Архитектурно:** кампании как master-процесс; отдельные документы заявочного контура; отдельный документ корректировки потребности; адресный маршрут согласования; новое РМ диспетчера; отдельный справочник бизнес-способов; отдельный документ плана потребности; маршрут утверждения с историей; отдельный документ закупочной заявки; новый документ и БП для согласования закупки. |

## План закупок (строки 24–26)

| Тема | Сводка |
|------|--------|
| **Допущения** | Носитель ПЗ — типовой `ПланЗакупок` УП; группировка по ТРУ — через `КодТРУ` и модель перечней (10); трассировка ID через `Назначения` при умеренном слиянии; статусы `СтатусыПланов` — источник «утверждён/не утверждён». |
| **Ограничения** | Не проверены все отчёты СКД на готовую группировку по перечням; объекты обмена с ИСУ Снабжение не идентифицированы по имени; не просмотрены все обработки формирования ПЗ; реквизит «ИдентификаторПотребности» глобально не найден — возможны другие имена; не трассировались полностью запреты в лотах/заказах при статусе ПЗ. |

| **Варианты реализации** | Расширить `ПланЗакупок` и обработки заполнения/выгрузки; отдельный документ программы с версией; ужесточить слияние и обязательность назначения; регистр соответствий с количественным разложением; расширить прикладные проверки и журнал смены статуса; БП согласования с задачами и ролями. |

| Тема | Сводка |
|------|--------|
| **Допущения** | Типовое замещение планов приемлемо при согласии юридической службы с терминологией; согласование корректировки может опираться на ту же статусную модель, что базовый план. |
